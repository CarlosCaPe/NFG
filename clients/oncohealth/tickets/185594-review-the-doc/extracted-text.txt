Objective
The NewUM Data Team provides the data architecture underpinning the NewUM prior-authorization
(PA) system for oncology and specialty-drug cases. The platform ingests operational data from
multiple upstream sources, applies medallion-layer (Bronze/Silver/Gold) transformations in
Databricks, and delivers analytically and operationally ready datasets to downstream consumers.
Success is measured when all ingestion pipelines run within defined SLAs, data quality gates
enforce zero silent propagation of bad data, downstream consumers receive certified gold datasets
on schedule, and the platform supports 10x current PA case volume without architectural change.
Requirements
• Ingest all registered data sources into Databricks Bronze Delta tables per documented SLAs.
• Implement medallion architecture (Bronze/Silver/Gold) with enforced DQ gates at each layer.
• Support near-real-time CDC ingestion from NewUM operational DB.
• Deliver Gold-layer datasets for analytic downstream.
• Enforce HIPAA-compliant PII tokenization at the Bronze write layer.
• Maintain data models and dictionaries for the Drugs, Eligibility and Provider domains.
• Provide bidirectional data exchange between NewUM and Databricks with latency appropriate
to each use case.
• Prohibit all analytic queries against operational databases; all analytics served from Databricks
Gold (or Silver where approved).
Success Criteria
• Eligibility and Provider data is available for NewUM.
• All ingestion pipelines meet defined Data-in-Bronze SLAs
(CDC: near-real-time; batch feeds: per schedule).
• Zero DQ failures propagate silently to Gold; quarantine rate < 1% per batch in steady state.
• Gold datasets available to analytic, BI, payer, and finance consumers within agreed freshness
windows.
• All schema changes follow the versioning and data contract protocol; no uncoordinated
breaking changes.

ingestion), Silver (cleansed/conformed), and Gold (business-facing) tiers. Apache Airflow on AKS
orchestrates all batch and hybrid pipelines. Unity Catalog governs access control, lineage, and data
discovery across all layers.

Payer Hierarchy
The schema encodes two converging hierarchies rooted at Payer. The product classification
hierarchy — MajorLineOfBusiness → LineOfBusiness → BenefitPlan — drives PDL lookups, NP
rule evaluation, and scope determination. Payer admin hierarchy — Payer → MajorLOB →
BusinessGrouping → BenefitPlan — carries admin-configured rules for scope, formulary, and NP
evaluation. Product is the convergence point for both, making it the structural anchor for all
downstream clinical and administrative rule resolution.
Member & Related Person
Person records are keyed by MemberID + Payer and serve as the PHI container — name, DOB,
gender, address, and related identifiers. RelatedPerson models the policyholder as a distinct entity,
while EmployerGroup captures group sponsorship context. ListOfMembers bridges the member side
to the eligibility core, likely supporting multi-member coverage units or dependent enumeration.
Eligibility Record
Eligibility records are versioned coverage spans linked to a Person and a Product/LOB classification.
The entity carries the full set of scope-determining attributes: AccountID, InsuranceGroupID,
BenefitPackage, CarrierID, ASO indicator, state of issue, member benefit flags, and
PrimaryOrCarveoutType. Scope validation evaluates all of these against the admin hierarchy; failure
on any check produces an Out-of-Scope determination, blocking clinical review. No case progresses
through member identification, coverage verification, or scope determination until all three steps
pass.

Versioning Strategy
All versioned tables use VersionCreatedAt (when this version becomes current) and VersionEnd
(when superseded, nullable - NULL means current). When a new version arrives for the same entity,
the ETL sets VersionEnd on the previous current row and inserts the new row. This is the one place
where an existing row is modified. DeletedAt serves a different purpose - it means the record was
deleted and should never surface in UI regardless of time queries. Both VersionEnd and DeletedAt
can be set on the same record.
Employer Group vs. Plan Taxonomy: Orthogonal Constructs
EmployerGroup, BusinessGrouping, and BusinessGrain are complementary but orthogonal
constructs. BusinessGrouping and BusinessGrain represent the payer's internal product taxonomy
— a stable classification hierarchy defining what kind of plan a product is. EmployerGroup
represents a specific employer's contractual relationship with the payer — who purchased coverage,
under what terms, and for how long. Two employers can purchase the same product (same
BusinessGrain) under entirely different contract terms, ERISA status, or external group identifiers;
the product taxonomy has no mechanism to capture this variation. Eligibility is the convergence
point, holding foreign keys to both, binding a member to a plan of classification and an employer
sponsor simultaneously.

Risks
The eligibility model is ~85% validated, but the remaining gaps — business hierarchy mapping,
manual overrides, and person identity — are structural blockers that affect rules of evaluation, scope
of determination, and multi-payer onboarding. Mid-case eligibility changes and inconsistent payer
file behavior add compliance and financial exposure. Dedicated workshops are planned to resolve
each item before implementation proceeds.
Manual eligibility overrides have no agreed storage pattern
Clinical staff frequently need to manually create or correct eligibility records (e.g., payer confirms
coverage by phone, but the file update takes two weeks). The Onco team and the NewUM team
have opposing positions on where this data should live. If manual records go into the ETL-managed
table, nightly batch loads will overwrite them. If they go elsewhere, the application needs to merge
two data sources at query time. Neither pattern has been designed.
Business impact: Without resolution, intake staff cannot process cases where eligibility data is
known-incorrect — a daily occurrence. Cases will stall or be processed against bad data, leading to
incorrect denials or approvals.
Working hypothesis: ETL-sourced and manually entered eligibility records live in the same table,
distinguished by a source flag (e.g., EligibilitySource: ETL | MANUAL). The ETL load logic is built to
respect this flag — batch loads only insert/version records where EligibilitySource = ETL and never
touch manually-created records. Manual records follow the same versioning pattern (immutable

Provider Core Identity - NPPES/Government Data
This is the anchor layer. The HealthcareProvider table is the root — every provider must exist here
before appearing anywhere else. It holds immutable attributes: ProviderId, NPI and entity type. From
here, the model branches into Doctor (Type 1) holding individual attributes like first name, last name,
gender, sole proprietor flag, and Facility (Type 2) holding organization name, facility type, and
related attributes. The key open question is whether these remain separate tables or merge into one
with an entity_type discriminator.
Provider Mutable State (Extensions of Blue Tables)
Attributes that change over time — name changes, credential updates, status changes. The meeting
determined these should not live in separate history tables but instead use single-table SCD Type
2 versioning (version_start, version_end, deleted_at), consistent with the eligibility domain. This
supports retrospective case lookups where you need to know what the provider looked like at case
creation time.
Taxonomy Classification - NPPES/NUCC Reference
Taxonomy codes classify a provider's specialty (e.g., cardiology, internal medicine). It's a many-to-
many relationship — one doctor can have multiple taxonomy codes. This requires a bridge table
between the provider and a taxonomy reference table. Both Type 1 and Type 2 providers can have
taxonomies. However, the group questioned whether taxonomy is even needed in the NewUM
application, since historically it's only been used for pre-UM filtering. Pending BA confirmation —
default position is to exclude from V1.
Address / Location (Shared Dimension)
Physical addresses for providers and facilities. The agreed direction is a shared address dimension
(junk dimension) reused across both the provider and eligibility domains. Key complexities: sole-
proprietor doctors have their own address; facilities have multiple locations; a doctor linked to a
practice does not inherit all practice addresses — they may only work at two of ten locations. This
requires bridge tables: T1-to-address and T2-to-address, with the linkage at the practice + specific
address level.
Payer-Submitted Provider Data (Green/Orange Tables — Client Data)
This is the provider data that payers send in their files. It includes the payer's own proprietary client
provider ID, their version of the provider's name, specialty, and address. This data must be stored
verbatim and returned on authorization files — payers expect exact round-trip fidelity. The grain of
the client provider ID varies per payer: some key by provider alone, others by provider + practice +
physical location.

NPPES-to-Client Linkage
The T1 Link and T2 Link tables join the NPPES reference data to the payer-submitted data using
NPI as the sole matching key. For V1, no fuzzy matching (name, specialty, zip code) is performed
— it's a straight NPI join. Match score and match method fields are unnecessary and will be
removed. This linkage is critical for enabling network checks and display enrichment, but the client's
original data is what flows back on auths.
ClientT1 - ClientT2 Practice Affiliation
This represents which doctor (Type 1) works at which facility/practice (Type 2). This linkage comes
exclusively from client data, not NPPES. The relationship is many-to-many and must be at the
address level — a doctor affiliated with a practice may only work at specific locations within that
practice. This is purely an employment/affiliation relationship, completely independent from network
participation.
Network Participation
The highest-volume and most complex category. Networks represent contractual agreements
between providers/facilities and payers. A single provider can belong to hundreds of networks (Cory
estimated 80M+ records). Key clarification from the meeting: network participation is independent
from T1-T2 affiliation. Both doctors and facilities independently link to networks. A shared network
dimension table holds the network definitions, with separate bridge tables linking T1-to-network and
T2-to-network. Networks drive contracting, claims payment, member benefit eligibility, and in/out-of-
network determinations.
Site of Care
Represents payer-preferred vs. requested treatment locations. A payer may deny a request at one
facility and redirect to a preferred lower-cost alternative. Currently implemented as a small Excel-
based dictionary for one client. BA requirements are still in progress. This category needs a formal
place in the model but its structure is deferred until requirements are finalized.
Risks
The provider domain carries risks concentrated in identity resolution, data volume, and unfinalized
design decisions. The highest-severity items — NPI-absent handling, network bridge scalability, and
unconfirmed versioning — are structural blockers to Silver schema finalization. A few key risks are
listed in detail below.
NPI-absent provider records have no agreed handling strategy
Providers submitted without NPI cannot be linked to NPPES reference data, which breaks network
checks, display enrichment, and downstream authorization workflows. The current Matis
workaround — setting NPI to all-nines or requiring manual entry — is fragile and does not scale to
multi-client onboarding. This affects every client that submits incomplete provider files.

Business impact: Cases referencing providers without NPI cannot complete network validation.
Intake staff must manually intervene, creating bottlenecks and increasing error rates on
authorization files returned to payers.
Working hypothesis: Provider records without NPI are loaded into the client provider table but
flagged with a source status (e.g., NPI_MISSING). These records participate in case processing
using client-provided attributes only. Network checks are skipped and the case is flagged for manual
review. NPPES linkage is not attempted. If the client subsequently provides the NPI, the record is
updated and linkage is retroactively established.
What this unblocks: ETL can load all client provider records without rejection. The application has
a clear rule: NPI_MISSING = manual network verification. No data is silently discarded.
What could invalidate it: If the volume of NPI-absent records is high for certain clients, the manual
review burden becomes unsustainable. Alternatively, if payers contractually require NPI on all auth
responses, records without NPI may need to be rejected at ingestion rather than loaded with a flag.
NPPES data currency is unverified
The system design designates NPPES as the provider master data source, the data may not be
consistently current. If NPPES records are stale, provider names, addresses, and specialties
displayed in the application could be incorrect. No one on the team has recently validated the
NPPES update cadence against actual data freshness.
Business impact: Clinical staff see outdated provider information during case processing. Incorrect
addresses affect place-of-service validation. Stale taxonomy codes could mislabel provider
specialties, undermining trust in the platform and creating audit findings.
Working hypothesis: NPPES weekly bulk files and incremental updates are ingested on a weekly
cadence into Databricks Bronze. A freshness check compares a sample of recently updated NPI
records against the CMS NPPES API to validate that the bulk file reflects current state. Freshness
metrics are tracked as a DQ dimension on the provider reference dataset.
What this unblocks: The team can proceed with NPPES as the reference source with confidence.
Freshness monitoring provides an early warning if data quality degrades.
What could invalidate it: If NPPES bulk files consistently lag real-world changes by weeks or
months, the platform may need a supplementary real-time lookup against the CMS NPPES API for
active case processing, adding infrastructure complexity.
Network participation data volume creates scalability risk
Each provider can link to hundreds of networks across clients. The network bridge could reach 80M+
records. The current proposed design with separate T1 and T2 tables doubles the bridge
infrastructure — separate network-doctor and network-facility bridge tables, each with their own
history. This is the largest data volume driver in the provider domain.

Business impact: Network checks are on the critical path for every prior authorization case. If the
network bridge is slow to query, case processing latency increases. Storage and compute costs

partitioned by payer_id and Z-ordered on provider_id + network_id for query performance.

What this unblocks: A single bridge table halves the maintenance surface. Partitioning and Z-
ordering keep network checks within latency SLAs. Volume validation provides a factual basis for
compute sizing.
What could invalidate it: If T1 and T2 remain in separate tables, the bridge must be duplicated. If
network participation data arrives as full-file replacements rather than deltas, incremental loading is
not possible and full reloads at 80M+ rows may exceed SLA windows.

Databricks Integration
The Cases domain is explicitly out of scope for the Data Team's medallion pipeline ownership, but
operational case data is a registered ingestion source. NewUM pushes case records to Databricks
Bronze on a daily batch cadence via the Data Team's REST ingestion API. The NewUM application
never reads from Databricks at query time for case processing — the Case schema in PostgreSQL
is the sole operational store. Downstream analytics consumers (case_summary, approval_funnel,
turnaround_time, case_audit_trail) are served exclusively from Gold.

Medalion Layer Design
The NewUM Data team owns delivery of raw NewUM data into the Databricks Bronze layer. From
there, the Onco Data team picks up the raw data and drives it through the medallion architecture —
Bronze → Silver → Gold — applying the cleansing, conforming, and certification steps described
below.
Once the Onco team publishes Gold-layer datasets purpose-built for NewUM, the NewUM team
consumes those certified tables and passes the data into the NewUM application. The medallion
layer descriptions below are included to paint the end-to-end picture; the transformation logic within
Silver and Gold is owned and executed by the Onco Data team.
Bronze Layer
Raw ingestion tier storing full-fidelity, append-only copies of all source events with minimal
transformation. Bronze is the system of record for replays, audits, and root-cause analysis. Each
CDC event is stored as a row with metadata columns (ingestion timestamp, source batch ID, event
schema version) appended by the ingestion framework.
Silver Layer
Transforms Bronze into a cleansed, deduplicated, conformed domain model. Applies type casting,
SCD Type 2 versioning, MERGE-based deduplication, referential integrity validation, and PII
tokenisation verification. Non-conforming rows route to quarantine
(silver.quarantine.<entity_name>). Key Silver transformations include: dosing JSON decomposed
into typed columns, CDC events deduplicated on PK + version key, and RI validated against Silver
dimension tables before MERGE.
Gold Layer
Purpose-built datasets for specific consumer use cases. Gold tables are certified, pass all DQ gates,
and are rebuilt on a scheduled cadence from Silver. Gold is the only layer directly accessible to BI
analysts, payer consumers, and external reporting tools.
Unity Catalog Structure
Data assets are organized by domain: payer (PDL, formulary, NP rules), drugs (DrugsMS catalog),
eligibility (payer feeds), providers (provider master data), and newum (operational case and drug
data). Naming convention: <domain>_<environment> (e.g., newum_prod, providers_staging). Final
naming to be confirmed during Unity Catalog governance setup.

Schema Evolution Approach
All schema changes should follow the Expand-Contract pattern. In Expand phase, new columns are
added as nullable with no default, or with backward-compatible defaults. Existing queries continue
to work. In Contract phase (after all consumers have migrated), NOT NULL constraints are added,
and old columns are deprecated. Enum additions require a migration file, a rollback plan, and sign-
off from the Schema Owner.
Metrics, Monitoring & Alerting
Data Quality
Six DQ dimensions are enforced: completeness (null rate thresholds), accuracy (domain/enum
validation), consistency (referential integrity), timeliness (lag monitoring), uniqueness (duplicate
detection), and validity (format/range checks). Tooling: Great Expectations for Silver entity suites,
dbt tests for Gold relational tests, Delta CHECK constraints as a storage-layer guard, and DLT
expectations where Bronze→Silver uses Delta Live Tables.
Quarantine & Alerting
Rows failing DQ at Silver are routed to quarantine tables (silver.quarantine.<entity_name>) with
failure metadata. Alert thresholds: >1% quarantined → P3 Slack; >5% → P2 PagerDuty; >10% or
PII-sensitive failure → P1 PagerDuty + incident creation. Quarantine rows retained 90 days; owning
engineer triages P1/P2 within 24 hours.
Job & Pipeline Monitoring
Each Databricks job is tagged (team, pipeline, environment, owner) for cost attribution. Performance
baselines (P50/P95 run duration, DBU consumption, rows processed) are established at launch;
>20% deviation triggers an alert. A centralized observability dashboard tracks pipeline run duration
trends, DQ pass/fail rates, quarantine trends, CDC lag, DBU cost by pipeline, and SLA breach
events.

Schema Versioning & Evolution
All Bronze schema changes are captured in the Avro Schema Registry. Compatible changes (new
nullable columns) are auto-applied via Auto Loader with mergeSchema=true. Breaking changes
(renames, type changes, drops) require advance notice, a Bronze migration script, Silver/Gold
migration jobs, and consumer notification before deployment. Silver and Gold schema changes
follow CI/CD with mandatory schema diff review. Delta table schema evolution is disabled by default;
only explicitly approved changes are applied. A schema changelog is maintained in /docs/schema-
history/ in the data platform repo.
Data Contracts
Every ingestion source must have a documented, owner-confirmed contract before production
onboarding. Contract formats: NewUM CDC uses JSON event envelope schema versioned in Git
(with optional Avro); DrugsMS uses OpenAPI spec with version field per response record; Payer
Eligibility uses payer-specific file specs with JSON schema in /docs/contracts/. Breaking-change
protocol: 7-day advance notice, coordinated cutover, and consumer notification.
Security and Compliance Concerns
The platform operates in a HIPAA-regulated healthcare prior-authorization context. PHI is tokenised
at the Bronze write layer before any downstream processing. The tokenisation service must support
the <5-minute CDC latency target. Unity Catalog enforces fine-grained RBAC including column- and
row-level security. ADLS Gen2 storage firewall policies, Azure Key Vault for secret management,
and Azure Active Directory for identity federation are core security infrastructure.
HIPAA BAA coverage is a mandatory filter for all third-party tooling evaluations (e.g., Fivetran,
Airbyte). Oncology data must not pass through third-party infrastructure without explicit BAA
verification.
Audit requirements: the gold.compliance.case_audit_trail table is stored in ADLS Gen2 immutable
storage (WORM) with append-only Delta, retained for 10 years. All Airflow DAG runs, Databricks
job executions, and data access events are logged and exported to Azure Monitor Log Analytics.
PHI field classification (beyond SSN, DOB, name) requires a data-classification review with
Legal/Compliance before the Silver schema is finalised. This is tracked as Open Question #9.
