Project introduction
Project Purpose and Background
The purpose of the NewUM project is to replace the end-of-life MATIS core monolith with a
modern, conﬁgurable, and high-performance Utilization Management (UM) case management
engine. The current MATIS system is based on a limitinGcode-based approach that prevents
eﬃcient customization, rapid client onboarding, and new feature development. NewUM will drive
operational eﬃciency and quality across OncoHealth by modernizinGarchitecture and data
modeling.
Project Goals and Objectives
● Improve client experience through better analytics (operational reportinGand
value-added insights for customers and internal OncoHealth teams) and provide a more
responsive client service experience when makinGneeded changes
● Improve the health care provider experience with a more responsive Portal and
extensibility of drug-based approval logiCinto provider-facinGtools
● Create an invisible migration experience for health plans and providers such that newUM
effort is perceived as an ‘upgrade’ of the system
● Improve OncoHealth Technical Team Eﬃciency by movinGto conﬁgurable architecture
and a new modern monolith
● Improve OncoHealth Case ProcessinGTeam eﬃciency (clinical and non-clinical) by
replacinGthe underlyinGabstraction from the MATIS platform (regimen-based
approvals) to drug-based approvals (potentially while retaininGthe concept of the
regimen) and signiﬁcantly improvinGthe UI experience with a cleaner set of clicks
● Create clear ‘hooks’ for AI agents to engage in workﬂow through design of APIs and UI
that are well-suited to current levels of agentiCAI capability

The backend architecture is split into three conceptual subsystems: Rules, Data and Interface.
Conﬁdential - Not for distribution Page 7

The Rules Subsystem will manage, compile and run sets and versions of rules. It will offer a
CLI-based development environment for conﬁguration specialists and interact with the Interface
subsystem via Azure Service Bus. It will use the data access layer as a library to interact with
cases.
The Data Subsystem will ensure consistency of data throughout NewUM. All access to
transactional data should go through the data access layer. It will offer a bulk ingest API for
migration and ingest pipelines, or it may be used as a library in other subsystems. All reporting
needs should be handled with an OLAP database speciﬁcally for reporting, populated by the
data access layer.
The Interface Subsystem will offer a uniﬁed interface to the UI and also peripheral services from
OneUM.
Our initial development will be focused on a vertical slice of the system that can create, view
and update a case as well as attach an eligibility record. The ﬁrst slice should also be able to
modify case state usinGbusiness rules. We will start with the green-highlighted components in
the Miro draft architectural diagram.
The data access layer is one of the most complex parts of the core. The technical demands for
trackinGcases are extensive:
- data for each case is spread across multiple tables
- some payers will want custom ﬁelds coupled with custom business rules
- maintain a complete audit loGfor each case, preferably with a non-PHI view for overseas
devs
- case data must be kept for years
- data will come from multiple sources: UI, external comms, ingest, Matis migration, etc.
- MS SQL database (OADB) is under strain in current system, while the business needs to
grow
- a case should enforce one writer at a time to prevent race conditions
The other schemas also have unique needs. The eligibility and provider schemas will need some
degree of masterinGto avoid confusion from duplicate provider and member records.
LeavinGthe core, NewUM will need to integrate with several peripheral services from OneUM like
a letter-sendinGsystem, a fax system, a document manager and so forth. Most of these other
services need upgrades (particularly for auth) and will require scaffoldinGso that the old and
new system can coexist. We should use this .NET package for consistency with other
OncoHealth systems unless there is a good reason not to.
