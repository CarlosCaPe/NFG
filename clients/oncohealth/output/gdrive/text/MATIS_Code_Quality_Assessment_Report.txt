CRITICAL SECURITY ISSUES 1. Path Traversal Vulnerability Severity:   CRITICAL Location:  AppealController.cs:158-177, AccountController.cs:303-322 Description:  File download endpoints accept user-controlled file paths with insufficient validation, potentially allowing attackers to access files outside intended directories. Code Example: public ActionResult OpenFile(string FilePath) {         if (!_baseManager.IsFilePathValid(FilePath))                 return RedirectToAction("NotFound");         FilePath = _parentDirectoryName + "\\" + FilePath;         if (System.IO.File.Exists(FilePath)) {                 return File(System.IO.File.ReadAllBytes(FilePath), ...);         } } Risk:  Attackers could read sensitive config files, credentials, or other users' documents. Recommendation: • Use file IDs instead of paths in URLs • Implement strict whitelist validation • Use Path.GetFullPath() and validate result • Add comprehensive audit logging 2. Async/Await Deadlock Risks Severity:   HIGH Location:  4 controllers using .Result, .Wait(), or .GetAwaiter().GetResult() Affected Files: • AssignProtocolController.cs • SiteOfCareController.cs • Saml2Controller.cs • SsoSaml2Controller.cs Risk:  Thread pool starvation, application hangs, poor performance under load. Recommendation: • Convert all blocking calls to async/await • Make controller actions async Task < ActionResult > • Use ConfigureAwait(false) where appropriate 3. JsonRequestBehavior.AllowGet on POST Methods Severity:   HIGH

FRONTEND CODE QUALITY ISSUES 1. Excessive Use of 'any' Type Severity:   MEDIUM Frequency:  303 occurrences across 75 TypeScript files Impact: • Loses IntelliSense/autocomplete • Runtime errors instead of compile-time errors • Difficult refactoring Recommendation: • Create interfaces for all models • Enable strict TypeScript mode • Use ESLint rule to prevent 'any' 2. Silent Error Swallowing in Frontend Severity:   HIGH Location:  http-error-handler.service.ts Description:  HTTP error handler silently swallows ALL errors and returns empty results. Code: handleError < T > (serviceName = "", operation = "operation", result = {} as T) {         return (error: HttpErrorResponse): Observable < T > => {                 //console.error(error); // Commented out!                 return of(result); // Returns empty!         }; } Impact:  Users get no feedback when operations fail. Recommendation: • Display error messages using toastr • Log errors to Datadog • Return error state instead of empty data 3. Observable Memory Leaks Severity:   MEDIUM Frequency:  229 subscribe() calls without unsubscribe Description:  Components subscribe without unsubscribing in ngOnDestroy. Recommendation: • Implement takeUntil(this.destroy$) pattern • Add ngOnDestroy to complete destroy$ subject • Use async pipe where possible
