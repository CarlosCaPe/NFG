The Job Monitor Service is a .NET 8.0 Console Application that monitors document mailing jobs and sends alert notifications (Email/SMS) when jobs fail to be transmitted or picked up by postal vendors within expected timeframes.

Type: Console Application (scheduled execution)
Framework: .NET 8.0
Purpose: Operational alerting for mailing failures

### 📠Fax Integration Overview & Proposal

newum_services &rarr; oh-fax-service Integration

Executive Summary

This document outlines the integration approach for connecting newum_services (.NET 10) with the existing oh-fax-service (Azure Functions) to enable inbound and outbound fax capabilities. We've identified critical security concerns with the current multi-tenant architecture and propose solutions to ensure proper data isolation.

Key Findings:

✅ oh-fax-service is a shared service used by 4 clients (MATIS, SmartUM, IRIS, newum)

⚠️ Security Gap: Inbound faxes are NOT isolated by ClientId

✅ Two viable architectural patterns identified

1. Current System Overview

1.1 Existing Architecture

oh-fax-service is a shared Azure Functions application that:

Sends faxes via OpenText API (third-party fax vendor)

Receives faxes via OpenText polling (every 5 minutes)

Stores metadata in SQL database [FaxService]

Stores PDF files in Azure Blob Storage

3. Problems Identified

3.1 Critical Security Issue: No Inbound Isolation

Problem:

The [FaxService].[InboundLog] table does NOT have a ClientId column. All clients query the same fax numbers.

Impact:

markdownwide760markdownwide760Result:

❌ MATIS users see newum's faxes

❌ NewUM users see MATIS's faxes

❌ Potential HIPAA violation (cross-client data exposure)

❌ Risk of linking the wrong fax to the wrong client

Why This Happens:

Faxes arrive at shared phone numbers (12292334086, 12292334097). There's no automatic way to determine: "This fax is for MATIS vs newum."

3.2 Architectural Pattern Inconsistency

Findings:

MATIS queries [FaxService] database directly (not via API)

oh-fax-service provides APIs (GET /getinboundfaxjobs), but MATIS doesn't use them

Unclear if NewUM should follow the MATIS pattern or use proper API-based microservices

3.3 Multi-Tenant Data Isolation

Current State:

✅ Outbound: Properly isolated by ClientId (works correctly)

❌ Inbound: Not isolated by ClientId (security gap)

Root Cause: Shared fax numbers + missing ClientId in InboundLog

4. Solution Options

Primary Approach: API-Only Integration ✅ (RECOMMENDED)

Approach: newum calls oh-fax-service APIs exclusively, no direct database access.

Architecture:

wide760What newum implements:

HTTP Client (OhFaxServiceClient) - calls oh-fax-service REST APIs

Business Service (CaseFaxService) - orchestrates fax operations

Single Database Table (CaseFaxLink) - links Cases to Faxes (see explanation in section 5.3)

API Controller (FaxController) - exposes endpoints to NewUM UI

What newum does NOT do:

❌ NO direct connection to FaxService database

❌ NO duplication of fax data (PDFs, metadata)

❌ NO background polling (oh-fax-service already does this)

Pros:

✅ Clean Microservices: Proper separation of concerns

✅ Loose Coupling: Schema changes in oh-fax-service don't affect newum

✅ Maintainable: API contract is stable and versioned

✅ No Database Credentials: Security benefit

✅ No Breaking Changes: Doesn't require modifying oh-fax-service

Cons:

⚠️ Security issue: Without additional measures (see options below), you can see other clients' unlinked inbound faxes

⚠️ HTTP overhead: Slightly slower than direct DB (negligible in practice)

Optional Enhancement: Dedicated Fax Numbers (Recommended for Security)

Approach: Purchase dedicated fax numbers for newum instead of sharing with other clients.

public async Task> GetInboundFaxJobs(
    string faxNumbers,
    int lastNdays,
    int clientId)
{
    return await _context.InboundLog
        .Where(f => f.ClientId == clientId)  // ← NEW
        .Where(f => faxNumbers.Contains(f.RecipientFaxNo))
        .Where(f => f.DeliveryTime >= DateTime.UtcNow.AddDays(-lastNdays))
        .Where(f => f.Downloaded == false)
        .ToListAsync();

Problem 3: Letter V2 Has Its Own UI

Letter V2 uses its own React-based UI app at letters.oncologyanalytics.com for the letter editor (variable review, editing, saving). NewUM will have its own React application as the frontend. We need to either:

Integrate with / embed the existing Letter V2 UI, or

Build our own letter editing flow, or

Skip the UI editor entirely (programmatic sending)

Problem 4: No Endpoint to Get Per-Template Variable Requirements

The notification service has ~120 possible variables globally. Each template uses only ~20-30 of them. Today, no API endpoint tells you which variables a specific template requires.

GetVariableConfigurationAsync &rarr; returns ALL ~120 definitions (no template filter)

GetTemplatesListAsync &rarr; returns template metadata (no variable information)

The per-template variable list exists internally in Cosmos DB (VariationCosmos.GetAllVariables()) but is not exposed via any API

Without this, we cannot programmatically validate whether we're providing all required variables before sending.

Problem 5: Template Selection Requires InsuranceProviderId Mapping

Templates are associated with InsuranceProviderId (int) from the OLTP system. NewUM uses Payer.Id (Guid). There is no existing mapping between these.

Problem 6: No Automatic Letter Sending Exists

The current system is 100% UI-driven. No service in the ecosystem automatically sends letters based on case decisions. There are no event listeners for case status changes that trigger letter creation. If NewUM wants automatic sending, it would be a new capability.

### Best Practice for Storage Accounts

Enable Secure Transfer

The secure transfer option enhances the security of your storage account by only allowing requests to the storage account by secure connection. For example, when calling REST APIs to access your storage accounts, you must connect using HTTPS. Any requests using HTTP will be rejected when 'secure transfer required' is enabled. When you are using the Azure files service, connection without encryption will fail, including scenarios using NFS v4.1, SMB 2.1, SMB 3.0 without encryption, and some flavors of the Linux SMB client. ### Branching Strategy and Merge Process

Iris uses Gitflow workflow which defines a strict branching model planned and designed around the project release. Gitflow is ideal for Iris since projects have a planned(scheduled) release cycle.

GitFlow WorkFlow

The overall flow (summary) of Gitflow is:

A develop branch is created from the main branch

A release branch is created from develop branch

feature branches are created from develop branch

When a feature is complete it is merged into the develop branch

When the release branch is done it is merged into develop and main branch

If an issue in the main branch is detected a hotfix branch is created from the main branch

Once the hotfix is complete it is merged to both develop and main

How it works

Develop and main branches

Instead of a single main branch, this workflow uses two branches to record the history of the project. The main branch stores the official release history, and the develop branch serves as an integration branch for features. It's also convenient to tag all commits in the main branch with a version number. This branch will contain the complete history of the project, whereas main will contain an abridged version. Other developers should now clone the central repository and create a tracking branch for develop.

Feature branches

Each new feature should reside in its own branch, which can be pushed to the central repository for backup/collaboration. But, instead of branching off of main, feature branches use develop as their parent branch. When a feature is complete, it gets merged back into develop. Features should never interact directly with main. Feature branches are generally created off to the latest develop branch.

Release branches

Once develop has acquired enough features for a release (or a predetermined release date is approaching), branch off of develop. Creating this branch starts the next release cycle, so no new features can be added after this point&mdash;only bug fixes, documentation generation, and other release-oriented tasks should go in this branch. Once it's ready to ship, the release branch gets merged into main and tagged with a version number. In addition, it should be merged back into develop, which may have progressed since the release was initiated.

Hotfix branches

Maintenance or &ldquo;hotfix&rdquo; branches are used to quickly patch production releases. Hotfix branches are a lot like release branches and feature branches except they're based on main instead of develop. This is the only branch that should fork directly off of main. As soon as the fix is complete, it should be merged into both main and develop (or the current release branch), and main should be tagged with an updated version number.

### Defect Creation Process

Defect Creation Process

This section defines the rules and steps for creating, linking, prioritizing, and managing defects to ensure consistency, traceability, and efficient resolution.

Defect validation

Before creating a defect, the QA Engineer verifies that the observed behavior represents an actual issue and not a misunderstanding, test data problem, or expected system behavior.

The QA Engineer performs the following validation steps:

Confirms that the issue is reproducible and supported by clear evidence.

Checks existing defects to avoid duplicates.

Ensures that the issue is not caused by environment instability or incorrect test setup.

Duplicate and Related Issues Handling

If the issue has already been reported, a new defect is not created.
Instead, additional findings or evidence are added as a comment to the existing defect.

If the issue is similar but not identical to an existing defect, the QA Engineer decides whether:

to add clarifying comments to the existing defect, or

to create a new defect with a clear description of the differences.

If the issue was previously reported and recently closed, and the same behavior is still present, the defect should be reopened with a clear explanation of the reason.

Defects closed long ago (e.g., related to earlier releases or milestones) should not be reopened.
In such cases, a new defect must be created and linked to the previous one if relevant.

ADO Defect Registration Rules

Defect Creation Rules

The following rules define how defects must be created, linked, and managed in Azure DevOps depending on the testing context and defect origin.

1. Defects Found During User Story Testing

If a defect is found during User Story testing and is directly related to that same User Story, the defect must be created as a Child Bug of the corresponding User Story.

If a defect is found during User Story testing and is related to another User Story from the current Sprint that is already closed, the User Story must be re-opened only for P1 and P2 defects that are directly related to that story.

If a defect is found during User Story testing and is related to a User Story from a closed Sprint, the defect must be created as a standalone Bug.

The bug is added to the current Sprint after priority clarification.

Defects with Low or Medium priority must be placed into the Backlog.

2. Defects Found During Regression Testing

If a defect is found during Regression testing, the defect must be created as a Child Bug of the Regression Test task.

The defect must be tagged with Regression.

3. Defects Found During UAT or Production

If a defect is found during UAT or in the Production environment, it must be created as a standalone Bug.

The defect must be tagged with:

UAT (for UAT environment), or

Production (for Production environment).

The defect is added to the current Sprint after priority clarification.

Defects with Low or Medium priority must be placed into the Backlog.

4. Test Traceability

All defects found during test execution must be linked to the corresponding Test Case(s) in Azure DevOps to ensure traceability between requirements, tests, and defects.

5. Re-opened Defects

If a defect that was closed in a previous Sprint becomes reproducible again, the defect must be re-opened, not re-created.

The re-opened defect must be linked to its parent User Story and added to the current Sprint.

### Deployment Process

The Iris release management process will follow a structured model with the following components at its core: Planning, Scheduling, Testing, and Deployment, allowing DevOps to control the project in each stage or environment. Following this approach will ensure application deployments are stable, smooth and avoid half-baked releases by constantly following a process and approval mechanism.

Development phase

The developer completes their sprint items and creates a pull request from the user feature branch to the develop branch. Upon creating a pull request, a build validation will be triggered automatically.

Upon a successful build, reviewers will review and approve the pull request only when all related build artifacts have been met. Once the DevOps build pipeline has finished, a release deployment build will be triggered and the components in the feature branch will be deployed to the Dev-Int sandbox.

Release phase

The successful completion of sprints items will be merged into the develop branch. A release branch will be created from the develop branch when higher environment deployments are needed.

Release Process Components

Release Model: A specific release process from project planning to delivery.

Release Policy: The definition of release standards, types, and requirements to set the release process

Release Template: A single and repeatable workflow to release process that includes human and automated activities. In other words, a release management checklist.

Release Plan: Activities involved in deploying a release to production

Major Release: Include many release units that have a higher and critical business impact

Minor Release: Include fewer release units that do not include critical components, such as breaking API changes, etc.

Key characteristics of a case include:

Documentation: Users may attach documents to an in-progress work session or a created draft case as part of the entry.

 | Eligibility refers to the system's determination of means whether a patient is entitled to receive specific healthcare services under a particular insurance plan or coverage program at a given time.

The Verification Process

Trigger: Verification occurs once member information is saved and a Treatment Start Date is provided.

Evaluation: The system compares the planned Treatment Start Date against the member&rsquo;s eligibility effective and expiration dates provided by the health plan.

Outcome: A member is considered "Eligible" if at least one eligibility record is active for the Treatment Start Date.

Audit: Every eligibility verification outcome is recorded for audit purposes.

### Test Case Creation, Review, and Execution Process

The QA testing process starts when a User Story is marked as Ready for QA. From this point, QA is responsible for preparing, creating, reviewing, and executing test cases, as well as coordinating defect handling according to the defined Defect Reporting Process.

At the beginning of the process, QA reviews the User Story and its Acceptance Criteria to ensure a clear understanding of the expected functionality. Based on this review, QA identifies the testing scope, potential risks, and dependencies that may affect testing activities.

As part of test preparation, QA creates the necessary QA-related tasks in Azure DevOps, such as Test Case Creation and, if required, Test Data Preparation. All QA tasks must be linked to the corresponding User Story to ensure traceability.

For each User Story, a dedicated Test Suite is created in Azure DevOps. The Test Suite must be named exactly the same as the User Story and assigned to the correct iteration. This Test Suite serves as a container for all test cases related to the given User Story.

QA then creates Test Cases within the Test Suite. Each Test Case must be linked to the related User Story and assigned a priority that reflects its importance:

P1 &ndash; covers the main workflow and core requirements of the User Story

P2 &ndash; covers additional workflows related to the User Story

P3 &ndash; covers major requirements not included in P1 or P2

P4 &ndash; covers remaining or edge-case requirements

Once test cases are created and considered ready for review, the Test Creation task is moved to the Closed state in Azure DevOps. Only after this step can the assigned QA proceed with the Test Demo and review activities.

The Test Demo is conducted by QA and must include the Developer who implemented the User Story, as well as other QA engineers and the BA/PO. During this session, test cases are reviewed, and feedback is collected.

If feedback requires updates to the test cases, the Test Creation task is re-opened (set to Active), and the necessary changes are applied to the test cases. After all feedback is addressed, the Test Creation task is closed again, and all related artifacts in Azure DevOps (test cases and test suites) are updated accordingly.

Following review and finalization, QA executes the test cases in the QA environment. During execution:

If a test case passes, it is marked as Passed.

If a test case fails, a defect is created and registered in Azure DevOps according to the Defect Reporting Process. The defect must be linked to both the User Story and the failed Test Case, and the Test Case is marked as Failed.

After execution, QA verifies whether all test cases have been executed. If there are failed test cases associated with open defects, QA waits for the fix to be deployed and then performs re-testing of the affected test cases.

If all test cases are passed and all related defects are either closed or accepted, QA work for the User Story is considered complete. The process ends with the confirmation that testing activities for the User Story have been successfully finished.

In cases of uncertainty or ambiguity in test results or expected behavior, QA escalates the issue to the BA/PO or QC Team Lead for clarification. Any risks identified during test case creation or execution are also raised for assessment and resolution.

### Testing Strategy

1. Introduction

 1.1 Purpose
The purpose of this document is to define the Test Strategy for the NewUM platform developed for OncoHealth.

This Test Strategy establishes the overall testing approach, principles, scope, and governance for all testing activities performed throughout the NewUM project lifecycle. It serves as a controlling document that ensures testing is planned, executed, and reported in a consistent, traceable, and standards-compliant manner, in alignment with ISO/IEC/IEEE 29119.

The primary objective is to move away from slow, manual verification toward a Continuous Testing model. By focusing 80% of our automation on the API and 20% on the UI, we achieve:

Faster Execution: API tests run in milliseconds compared to seconds for UI.

Earlier Bug Detection: Finding a logic error in an API test is significantly cheaper than finding it during a full E2E UI run.

Reduced Flakiness: API responses are deterministic; UI tests can fail due to network lag, browser updates, or "race conditions."

2. Testing Methodology

2.1 Test Levels

Testing of the NewUM platform is performed across multiple test levels, each addressing specific quality objectives and risks. The defined test levels align with ISO/IEC/IEEE 29119 and support the phased delivery and migration strategy adopted by OncoHealth.

Each test level has a clearly defined purpose, scope, and responsibility, and collectively they provide comprehensive coverage from individual components to full system operation.

Unit Testing (Level 1)

Responsibility: Developers.

Focus: Isolated logic (e.g., the function that case was created or updated according to input parameters in method).

Tool: .

API & Integration Testing (Level 2 - The Priority)

Responsibility: Developers (base integration testing) and QA Team (extended negative scenarios and flows).

Focus: Validating that the "Engine Room" (Business Logic) works.

Strategy: Test all permutations here

Tool: Playwright request fixture with .NET.

UI E2E Testing (Level 3 - The Sanity Check)

 Responsibility: QA Team.

Focus: User experience and "wiring."

Strategy: Testing only the "Happy Path" here.

Tool: Playwright + Typescript.

2.2 Automation API Test Strategy (The &ldquo;Engine Room&ldquo;)

All permutations of business logic will be tested here to save time and increase stability.

Positive Scenarios: Valid inputs, successful CRUD operations.

Negative Scenarios: Invalid data, missing headers, 400/500 error handling.

Security (RBAC): Using the scopesByDomain structure to ensure unauthorized users receive 403 Forbidden.

Performance: Verify response times for heavy endpoints (e.g., CaseCreation) are under defined time.

2.3 Automation UI Testing Strategy (&ldquo;Happy path&ldquo;)

UI tests will be kept "thin" to avoid flakiness and slow build times.

Critical Path Testing: Only automate flows that lead to "Value" (e.g., Login, Creating a Case, Searching, Viewing a Report).

Browser Coverage: Primary focus on Edge, with sanity checks on Webkit (Safari) and Chromium.

State Injection: Do not use the UI to set up data. Use API calls to "prime" the system state before the UI test runs.

The scope reflects the phased delivery model, parallel operation with the legacy MATIS system, and the regulatory context of healthcare software

functionality not included in the approved NewUM scope

4. Test Process

This section describes the testing process lifecycle applied to the NewUM platform. The process follows ISO/IEC/IEEE 29119 principles and defines how testing activities are planned, prepared, executed, and completed in a controlled and repeatable manner.

Testing activities are integrated into the project lifecycle and adapted to the phased delivery approach.

4.1 Test Planning

Test planning defines how testing activities are organized, prioritized, and aligned with project delivery for the NewUM platform. Planning ensures that testing is focused on the most critical areas, is feasible within project constraints, and provides meaningful input to delivery and release decisions.

Test planning is performed at the beginning of each project phase, release, or significant change, and is refined continuously as scope, risks, or requirements evolve.

During test planning, the following activities are performed:

Review of inputs
Approved requirements, user stories, acceptance criteria, configurations, and architectural decisions are reviewed to understand expected system behavior and identify testing needs.

Scope and priority definition
The scope of testing is defined for the given phase or release, with priorities assigned based on business impact, data sensitivity, integration complexity, and regulatory risk.

Risk assessment
Key risks related to functionality, data migration, integrations, performance are identified and used to guide test coverage and depth.

Test approach selection
Appropriate test levels, test types, and execution methods (manual or automated) are selected based on system maturity and risk profile.

Resource and dependency planning
Required environments, test data, tools, and roles are identified, along with dependencies on other teams or systems.

Alignment with delivery milestones
Testing activities are aligned with development timelines, release schedules, and acceptance checkpoints to ensure timely feedback.

Test planning outputs include defined test scope, identified risks, agreed priorities, and a shared understanding of testing expectations for the upcoming work.

4.2 Test Design and Implementation

Test design and implementation define how test scenarios and test cases are created, organized, and maintained to validate the NewUM platform.

Test design begins once requirements, user stories, configurations, or acceptance criteria are sufficiently defined. The focus is on translating business and technical expectations into clear, traceable, and executable test cases.

During test design and implementation, the following activities are performed:

Test scenario identification
Test scenarios are derived from user stories, acceptance criteria, configurations, and key workflows. Scenarios reflect real user behavior, critical paths, and identified risk areas.

Test case creation
Test cases are created with clearly defined preconditions, steps, and expected results. Each test case is written to be atomic, repeatable, and understandable without additional context.

Test organization and traceability
Test cases are organized into test suites and linked to corresponding user stories or requirements to ensure traceability and coverage visibility.

Test data preparation
Required test data is identified and prepared to support execution of test cases, including representative data sets for functional, integration, and migration scenarios.

Automation consideration
During implementation, test cases are reviewed to identify candidates for automation. 

Review and maintenance
Test cases are reviewed for clarity, correctness, and completeness. Existing test cases are updated as requirements or configurations change to ensure continued relevance.

Test design and implementation are continuous activities and evolve alongside the system as functionality is added or modified.

4.3 Test Environment Preparation

Test environment preparation ensures that testing is performed on test environment.

In general, test environment preparation is expected to include:

validation of environment availability and stability;

verification of configuration and version alignment with the target release;

confirmation of user roles, permissions, and access controls;

preparation of test data in accordance with security and compliance requirements;

coordination with dependent systems or external integrations, where applicable.

Environment readiness will be confirmed prior to test execution to ensure reliable and consistent test results.

4.4 Test Execution

Test execution is the process of running planned test cases to validate the behavior of the NewUM platform and to collect objective evidence of system quality.

Test execution is performed once test planning, test design, and environment preparation activities are sufficiently completed. Execution activities follow the defined test scope and priorities for the given phase, release, or change.

During test execution, the following activities are performed:

execution of test cases according to the approved test scope and schedule;

recording of test results, including pass, fail, blocked, or not applicable statuses;

collection of supporting evidence such as screenshots, logs, or output data;

identification and reporting of defects for failed test cases;

re-testing of fixed defects to verify resolution;

execution of regression tests as changes are introduced.

Test execution progress and outcomes are tracked to provide visibility into quality status, risks, and readiness for the next delivery stage.

4.5 Test Completion

Test completion defines the activities performed to formally conclude testing for a given phase, release, or defined scope of work.

Test completion is initiated once planned test execution activities are finished or when exit conditions for the testing scope are met.

During test completion, the following activities are performed:

confirmation that planned test cases within the defined scope have been executed;

verification that acceptance and exit criteria are satisfied;

review of open defects and confirmation that critical issues are resolved or formally accepted;

assessment of residual risks associated with known limitations or deferred defects;

documentation of test results and overall testing outcomes;

communication of testing status and recommendations to relevant stakeholders.

Test completion provides objective evidence to support release decisions, phase transitions, or client migration readiness.

5. Test Environments

5.1 Environment Description

The Test Environment serves as a dedicated, stable staging area where integrated components (API, UI, Database) are verified prior to production.

Infrastructure: A cloud-based replica of the production stack (Azure). This includes the API services, the frontend hosting, and the database instances.

Integrations: The environment is connected to test instances of our messaging bus (Azure Service Bus) to ensure event-driven flows are fully exercised.

Accessibility: Accessible via the VDI for development and QA teams, with automated access granted to the CI/CD runners.

Consistency: To ensure deterministic results, the environment version matches the "release candidate" branch.

5.2 Environment Constraints

Constraints are the "known limitations" that the team must account for to avoid false negatives in testing.

Resource Scaling: The Test Environment uses smaller instance sizes compared to Production. Consequently, performance testing results in this environment should be viewed as relative, not absolute.

External Dependencies: Third-party vendors could have Mocks or Stubs to avoid costs and prevent test failures caused by external downtime.

Concurrency Limits: While the environment supports parallel execution of tests, there is a cap on the number of concurrent browser instances to prevent CPU throttling.

Data Persistence: This environment is subject to a "Clean Slate" policy. Databases may be wiped and re-seeded weekly or per-release to prevent data bloat and configuration drift.

5.3 Test Data Requirements

Since our application processes health-related data, the strategy for test data is critical for both realism and security.

PHI-Similarity (Synthetic Data): We will use Synthetic Data Generation rather than real patient records. This data must maintain "Clinical Realism"&mdash;meaning it follows the same patterns as PHI (Names, DOBs, Social Security formats, ICD-10 codes) but contains no real-world individuals.

Data Masking/Anonymization: If any data is pulled from higher environments for debugging, it must go through a De-identification process to remove all 18 HIPAA identifiers before reaching the Test Environment.

Seeding Mechanism: * Static Data: Standard configurations (like scopesByDomain settings) are loaded via SQL scripts during environment setup.

Dynamic Data: Automation tests will use API POST requests to generate "Fresh" patients and cases for each test run. This ensures that every test starts with a known, predictable state.

Volume: For major API scenarios, the data set must include edge cases: patients with very long medical histories, records with special characters, and missing optional fields to verify system robustness.

9. Entry and Exit Criteria

This section defines the conditions that must be met before testing can begin and the conditions required to formally complete testing for a given phase, release, or scope.

Entry and exit criteria help ensure that testing is performed on a stable and meaningful baseline and that results can be used to support informed delivery and release decisions.

9.1 Entry Criteria

Testing activities may begin when the following conditions are met:

requirements, user stories, and acceptance criteria for the testing scope are defined and approved;

scope of testing for the phase or release is agreed;

test cases are created and reviewed for the defined scope;

test environments required for execution are available and accessible;

required test data is prepared or available;

dependencies on external systems are identified and addressed;

known blocking issues are resolved or accepted.

If entry criteria are not fully met, testing may proceed only with documented risks and stakeholder agreement.

9.2 Exit Criteria

Testing activities may be considered complete when the following conditions are satisfied:

planned test cases for the defined scope are executed;

test results are recorded and reviewed;

critical and high-severity defects are resolved or formally accepted;

no unresolved blocking issues remain;

regression testing is completed for affected areas;

residual risks are identified and communicated;

testing outcomes support release, phase transition, or migration readiness decisions.

Exit criteria are evaluated in the context of risk and business impact rather than solely on defect counts.

1. What Is Dynamic LINQ

Dynamic LINQ (NuGet: System.Linq.Dynamic.Core) is an open-source library that extends .NET LINQ by allowing queries to be expressed as strings rather than compile-time lambda expressions. It parses a text condition at runtime and produces a standard System.Linq.Expressions.LambdaExpression &mdash; the same expression tree the C# compiler generates from a => lambda.

Why It Matters

In conventional LINQ, conditions are fixed at compile time:

c#wide760 p.Age >= 18);]]>With Dynamic LINQ, the condition can come from configuration, a database, or a JSON file:

c#wide760= 18");]]>This makes Dynamic LINQ ideal for scenarios where business rules change frequently and must be modified without recompiling the application &mdash; rule engines, configurable filters, admin-driven query builders, and reporting tools.

5. Building NRules Rules from JSON with Dynamic LINQ

NRules is an open-source .NET rule engine based on the Rete algorithm &mdash; a pattern-matching algorithm optimized for evaluating many rules against a shared set of facts. By combining NRules with Dynamic LINQ, we can define rules in JSON and compile them into a Rete network that evaluates efficiently.

NRules Fundamentals

NRules works with facts (objects inserted into a session) and rules (LHS conditions + RHS actions):

wide760 0     │

│             THEN RecordResult(...)              │
│     ...                                         │
│                                                 │
│   session.Fire()  →  matched rules execute      │

    public Dictionary> EvaluateAll(RuleContext context)
    {
        var collector = new RuleResultCollector();

        // Create session, insert facts, fire rules
        var session = _sessionFactory.CreateSession();
        session.Insert(context);
        session.Insert(collector);
        session.Fire();

        // Build complete results including unmatched and errored rules
        var results = new Dictionary>();
        foreach (var ruleSet in _config.RuleSets)
        {
            var ruleSetResults = new List();
            foreach (var rule in ruleSet.Rules)
            {
                // Check for compilation errors
                var errorResult = _compilationErrors
                    .FirstOrDefault(e => e.RuleId == rule.Id && e.RuleSetName == ruleSet.Name);
                if (errorResult != null)
                {
                    ruleSetResults.Add(errorResult);
                    continue;
                }

                // Check if the rule fired
                var firedResult = collector.Results
                    .FirstOrDefault(r => r.RuleId == rule.Id && r.RuleSetName == ruleSet.Name);
                if (firedResult != null)
                {
                    ruleSetResults.Add(firedResult);
                }
                else
                {
                    // Compiled OK but condition was false
                    ruleSetResults.Add(new RuleResult
                    {
                        RuleSetName = ruleSet.Name,
                        RuleId = rule.Id,
                        RuleName = rule.Name,
                        Matched = false,
                        Action = rule.Action,
                        Severity = rule.Severity,
                        Message = rule.Message
                    });
                }
            }
            results[ruleSet.Name] = ruleSetResults;
        }
        return results;
    }

c#wide760Why NRules + Dynamic LINQ?

The Rete algorithm provides advantages over simple sequential evaluation:

Shared sub-expressions &mdash; If multiple rules check Eligibility.IsActive, the Rete network evaluates it once and shares the result across all dependent rules.

Incremental evaluation &mdash; When facts change, only affected rules are re-evaluated.

Forward chaining &mdash; Rules can insert new facts that trigger other rules.

Conflict resolution &mdash; NRules provides built-in strategies for ordering rule execution.

Dynamic LINQ serves as the condition parser &mdash; it takes the JSON string and produces the LambdaExpression that NRules needs for its pattern conditions. Without Dynamic LINQ, each condition would need to be hand-coded as a C# expression or parsed by a custom parser.

Start with the Dynamic approach if you need a quick proof-of-concept or your rules change per-request.

Move to the Strong-Typed approach when you need production performance, expression tree inspection, or type-specific rule sets that evaluate against a single domain object.

Microsoft RulesEngine is an open-source .NET library (available as a NuGet package) designed to abstract business logic away from your core application code. Instead of hard-coding complex if-else or switch statements, you define your rules in a JSON format. This allows you to update your business logic dynamically without having to recompile or redeploy your entire system.

By defining logic in JSON, clinical staff or policy experts can potentially update rules in a database or configuration file while the application remains running.

How it Works

The engine evaluates data against a set of predefined conditions.

Workflow: A high-level process, such as "PatientAdmission" or "InsuranceClaimValidation."

Rule: Specific clinical criteria (e.g., "If Age > 65 and HeartRate > 100, flag as High Risk").

Scoped Parameters: Reusable variables, like a "CriticalThreshold" that can be adjusted globally for different hospital wards.

Lambda expression support &mdash; rules use C#-like expressions under the hood (powered by System.Linq.Dynamic.Core)

Nested rules &mdash; supports AND/OR groupings with child rules for complex clinical criteria

Custom actions &mdash; trigger alerts, notifications, or workflow transitions

Local params &mdash; compute intermediate clinical scores (BMI, eGFR, risk scores)

Scoped params &mdash; share computed values across rules in a workflow

Healthcare-Specific Benefits

Regulatory compliance &mdash; update rules when CMS, FDA, or clinical guidelines change without code deployments

Audit trail &mdash; rules stored as JSON provide clear documentation of clinical decision logic

Clinical staff ownership &mdash; physicians and compliance officers can review/modify rules in a familiar format

Multi-tenant flexibility &mdash; different insurance plans or hospital systems can have different rule sets

Considerations

Rules are evaluated using dynamic compilation, so there's a small performance overhead compared to hardcoded logic

Complex clinical protocols may require careful rule ordering and conflict resolution

HIPAA considerations &mdash; ensure rule definitions don't contain PHI

No built-in UI for rule management

Important: Rules engines are not suitable for controlling workflows, including Microsoft RulesEngine.
In our case, they should be treated as execution rules evaluated within a given workflow state.

Neither NRules nor Microsoft RulesEngine provide adequate support for defining workflows or state machines.
They are effective for simple rule execution at specific steps of a Case, but not for modeling or managing workflow lifecycles.

    {
        while (!IsTerminalState(context.CurrentState))
        {

            await ExecuteStateEntryActionsAsync(context);

                if (autoTransition != null)
                {
                    await ExecuteTransitionAsync(context, autoTransition);
                    continue;
                }

                // Pause for manual action
                return ProcessingResult.PausedForManualAction(context.CurrentState);
            }

            // 4. Evaluate transitions for automatic states
            var transition = await EvaluateTransitionsAsync(context, stateConfig);

            if (transition == null)
            {
                return ProcessingResult.NoValidTransition(context.CurrentState);
            }

            // 5. Execute transition actions and move to next state
            await ExecuteTransitionAsync(context, transition);
        }

        return ProcessingResult.Completed(context.CurrentState);

wide760Auto-Approval Implementation

Auto-approval occurs when all eligibility criteria are met:

c#wide760 ExecuteAsync(
        CaseStateContext context,
        ActionParameters? parameters = null)
    {
        var scopeResult = context.ScopeResult;
        var payerConfig = context.PayerConfiguration;
        var programConfig = context.ProgramConfiguration;

        // Check 1: Payer enables auto-approval
        if (!payerConfig.EnableAutoApproval)
        {
            scopeResult.AutoApprovalDecision = false;
            scopeResult.AutoApprovalReason = "Auto-approval not enabled for payer";
            return ActionResult.Success();
        }

        // Check 2: Has in-scope items requiring PA
        var inScopeItems = scopeResult.ItemResults
            .Where(i => i.ScopeClassification == ScopeClassification.PA_Required_InScope)
            .ToList();

        if (!inScopeItems.Any())
        {
            scopeResult.AutoApprovalDecision = false;
            scopeResult.AutoApprovalReason = "No in-scope items requiring PA";
            return ActionResult.Success();
        }

        // Check 3: All in-scope items eligible for auto-approval
        if (!inScopeItems.All(i => i.IsEligibleForAutoApproval))
        {
            scopeResult.AutoApprovalDecision = false;
            scopeResult.AutoApprovalReason = "Some items not eligible for auto-approval";
            return ActionResult.Success();
        }

        // Check 4: Match auto-approval criteria
        var criteria = programConfig.AutoApprovalCriteria;
        var allItemsMatchCriteria = inScopeItems.All(item =>
            criteria.Any(c => c.RxCuiCodes.Contains(item.RxCui)));

        if (!allItemsMatchCriteria)
        {
            scopeResult.AutoApprovalDecision = false;
            scopeResult.AutoApprovalReason = "Items do not match auto-approval criteria";
            return ActionResult.Success();
        }

        // All checks passed - eligible for auto-approval
        scopeResult.AutoApprovalDecision = true;
        scopeResult.AutoApprovalReason = "All criteria met for auto-approval";

        return ActionResult.Success();
    }

c#wide760 ContinueWithManualActionAsync(
    CaseStateContext context,
    ManualAction action)
{
    // Validate action is allowed in current state
    var stateConfig = _configService.GetStateConfig(context.CurrentState);

    if (!stateConfig.ManualActions.Contains(action.ActionType.ToString()))
    {
        throw new InvalidOperationException(
            $"Action {action.ActionType} not allowed in state {context.CurrentState}");
    }

    // Find and execute the matching transition
    var transition = stateConfig.Transitions
        .FirstOrDefault(t =>
            t.TransitionMode == TransitionMode.Manual &&
            _conditionEvaluator.Evaluate(t.Condition, context, action));

    if (transition == null)
    {
        return ProcessingResult.NoValidTransition(context.CurrentState);
    }

    // Execute transition actions
    await ExecuteTransitionActionsAsync(context, transition.Actions);

    // Move to next state
    context.CurrentState = Enum.Parse(transition.ToState);

    // Continue processing
    return await ProcessAsync(context);

wide760Summary

This rules engine implementation provides:

Flexible Rule Definition: Business rules defined in JSON, not code

State Machine Workflow: Clear state progression with defined transitions

Dual Approval Paths: Automatic and manual approval mechanisms

Condition Evaluation: Dynamic expression evaluation for transitions

Action Pipeline: Pluggable actions executed at each state

Audit Trail: Complete history of state transitions and manual actions

Extensibility: Easy to add new states, rules, and actions

The separation of concerns between rule definition (JSON), rule evaluation (Rules Engine), state management (State Machine), and action execution (Action classes) makes the system highly maintainable and configurable.
