## PERSEUS

Precision Evaluation & Reporting System for Enterprise User Statistics

 Purpose

What is PERSEUS?

Perseus is an automated reporting generation and validation system for OncoHealth&rsquo;s monthly and quarterly client deliverables.

PERSEUS Allows us to:

Automatically curate the data needed for each reporting deliverable

Schedule the reports to be created and land in a central location for user review

Conduct enhanced data validation across these reports in compliance with expected output formats

Move files from QA to Production for enterprise access without unnecessary PHI duplication on local machines

Project Artifacts:

The code used to Operate PERSEUS processing can be found in the reporting_fortify repo in Azure Dev Ops here: https://oncologyanalytics.visualstudio.com/Technology/_git/reporting_fortify

### ADOS - Tips on Writing Work Items

Overview

falseThis page has few tips to follow when a Work Item is created in ADOS.

Description

black1solidIn order to  make use of the ADOS board very efficiently, please make sure to follow all the important criteria while creating a new work item in ADOS.

Going forward we expect the team to start writing User Stories and Acceptance Criteria. This will help us in organizing our work and fine-tune the processes.

The ADOS tips are classified under the following topics

Work Items in ADOS

black1solid

PO: Product Owner

black1solidWork Items Hierarchy in ADOS 

black1solidImportant Points to Remember

Any Feature or User Story created must be placed under an EPIC.

All the EPICS are a direct pull over from our OKR document.

If not required, You can skip creating Feature and directly create an User story under an EPIC.

Though you can create a Task under Feature, it will show as UNPARENTED while filtering. So always make sure to create an User Story and then add the Tasks under it.

Moreover it is not possible to filter by Tasks in the ADOS Board as well as in the Backlog view.

Always break down your User Stories into many Tasks.

Estimated time for each task is the key in arriving Story Points for the User Story.
Always add the Story Points (Refer  for Story points)

Title for Work Item

black1solidDo not write a sentence for a title.

Use the description section to provide a detailed explanation of the issue being reported.

Do not use a prefix in your title unless instructed to do so (i.e. a policy agreed to on the team)

Description of a Work Item

black1solidNote: Never leave the Description portion blank when you create an EPIC, Feature, User Story or Task.

Work Item | Description | Acceptance Criteria Needed | 
Epic | Description &ndash; Major User Story. Few sentences explaining the EPIC | No | 
Feature | Description &ndash; Write about the Feature explaining it in detail | No | 
User Story &ndash; Individual Deliverable | Description &ndash; write User Story here | Yes | 
Task | Description &ndash; Explain the Task you are going to do | No | 

User Story

After reading a user story, the team knows why they are building what they're building and what value it creates.           

black1solidUser stories are

Short - expressed in a few short sentences, ideally using non-technical language.
Simple descriptions of a feature told from the perspective of the person who desires the new capability, usually a user or customer of the system
User stories are one of the core components of an agile program. They help provide a user-focused framework for daily work &mdash; which drives                                   
                                   *Collaboration

                                   *Creativity

                                   *Better product overall

Sample Template for an User Story

black1solidAs a < type of user >, I want < some goal > so that < some reason >

Example:As <a data analyst>, I want <to do an evaluation and distribution analysis of the Gateway claim files> so that < I can do a pre-analysis data review>

Acceptance Criteria 

All User Stories must have the Acceptance criteria written.

Clearly written criteria introduce a single solution to the functionality you intend to implement.

Creating acceptance criteria is all about gaining a common understanding. It requires the team to think about the proposed acceptance criteria, and then bring their questions and thoughts to the table with the product owner.

black1solidWhat are Acceptance Criteria Used For

To define boundaries. 

To reach consensus. 

To serve as a basis for tests.(QA)

To allow for accurate planning and estimation.

Acceptance Criteria Format Types and Structures

black1solidAcceptance Criteria can be written in different formats.

There are two most common ones, and the third option is to devise your own format:

Scenario-oriented (Given/When/Then)

Rule-oriented (checklist)

Custom formats

Scenario - Oriented

Scenario-oriented format of writing Acceptance Criteria is known as the Given/When/Then (GWT) type.

Given some precondition

When I do some action

Then I expect some result

black1solidScenario &ndash; the name for the behavior that will be described

Given &ndash; the beginning state of the scenario

When &ndash; specific action that the user makes

Then &ndash; the outcome of the action in &ldquo;When&rdquo;

For User Story Example: As a <Data Analyst> I want to <Filter work items in ADOS based on XYZ criteria> so that <I can generate a performance report for the Adhoc team>

Scenario - Ability to filter by XYZ criteria requested by the analyst in Power BI

Given - All the Analyst requested XYZ filter criteria in Power BI

When - Analyst chooses a specific filter

Then - Analyst is able to filter by chosen criteria

Rule - Oriented

black1solidThe rule-oriented form entails that there is a set of rules that describe the behavior of a system. Based on these rules, you can draw specific scenarios.

Usually, criteria composed using this form look like a simple bullet list. 

For User Story Example: As a <Data Analyst> I want to <Create a work flow in Decisions> so that <I can do XRT savings analysis>

Create a XRT savings calculation Simple Flow in Decisions

Create and add required Simple Flow steps in Decisions

Create and add required Forms in Decisions 

Test the XRT savings calculation Simple Flow

Custom Format

You can invent your own acceptance criteria given they serve their purposes, are written clearly in plain English, and can&rsquo;t be misinterpreted. 

black1solidFor User Story Example: As a <ADOS admin> I want to <Create a Confluence Page > so that <I can publish few important ADOS tips while creating a new Work Item to the team>Create a new Confluence page adding all important ADOS tips and publish it to the team so that they can follow the tips easily while creating their new Work Items.

Add the new page to the Analytics Team space for easy access by the team.

black1solidFew Tips to Write Great Acceptance Criteria

Keep your criteria well-defined so any member of the project team understands the idea you&rsquo;re trying to convey.

Keep the criteria realistic and achievable. 

Define the minimum piece of functionality you&rsquo;re able to deliver and stick to it. On the other hand, don&rsquo;t try to describe every detail since you risk cluttering up your backlog and getting buried under many small tasks.

Coordinate with all the stakeholders so your acceptance criteria are based on consensus.

Create measurable criteria that allow you to adequately estimate development time so you&rsquo;re able to stay within budget and time constraints (this is important in determining the Story points for Sprints)

Consider providing checklists that enable you to see what user stories are covered with acceptance criteria.

Attachments and Links to Work Items and Check History

black1solidMake sure you attach all files (Word, Excel, PPT etc. you created) and links related to the work item. 

This helps in understanding the work items better as well as saves time if someone is on PTO (we don&rsquo;t need to wait for the person to get the files)

### Clinical Data Repository (CDR)

Overview

falseThe Clinical Data Repository (CDR) is a consolidated repository of all Oncology Analytics client data, as well as an extensive list of publicly available reference data. This data is compiled in a way that makes it easier to do statistical analysis and analytical modeling. 

The CDR uses the Observational Medical Outcomes Partnership Common Data Model (OMOP CDM) as the basis for it's data storage. From there, a flattened data-set is available for use in any modeling efforts. 

Clinical Data Repository (CDR) Mission

falseThe Oncology Analytics (OA) clinical data repository (CDR) implementation requirements supporting the ability to make real world data readily available for life sciences research, clinical decision-making and pharma go-to-market acceleration include the following characteristics:

Industry Standard Data Model
Comprehensiveness
Regulatory Compliance

Industry Standard Data Model

The OA CDR must be able to overcome semantic challenges arising from the use of different languages, data models and systems.  Interoperability among clinical, administrative and other third-party systems is the core functionality required to aggregate and use patient data from disparate sources.  Interoperability eliminates the bottlenecks presented by separate systems, different data models, different coding and content standards, different usage of structured (e.g. laboratory results) and unstructured (e.g. clinician notes) data, and different languages.  Real-world data from EMRs and other clinical systems can be shared using a wide range of interoperability languages such as HL7 &reg;, FHIR&reg;, CDA, and many more. Clinical research systems must be able to connect to these applications in their native interoperability languages to effectively ingest all the desired data. In its source system, the clinical data may adhere to a specific data standard, such as HL7 Clinical Document Architecture (CDA). The OA CDR will leverage both a distributed file system-based data lake to ensure the ability to land any data regardless of the source system, with available open-source tools for speed and flexibility with respect to data transformation, mapping and ingestion into a standard data model.  All data is ultimately converted to the OMOP (Observational Medical Outcomes Partnership) Common Data Model (CDM) which has become a leading industry standard and a top choice for outcomes-based research on real world data (Garza, Del Fiol, Tenenbaum, Walden & Zozus, 2016).

Comprehensiveness

The OA CDR must ensure data quality and completeness with the ability to evaluate structured and unstructured data.  All real-world data must be connected, comprehensive, and credible to ensure fitness for its intended purpose and confidence in the results. Additionally, for life sciences and pharma observational investigations, the data must be current. Interoperability addresses the issue of connected, standardized data.  The issue of comprehensive data is addressed by the OA CDR through its ability to aggregate heterogeneous health information from multiple sources, including medical claims, pharmacy claims, laboratory result feeds, NLP medical chart and clinical note extraction, and prior authorization transactions.  The OA CDR will be able to ingest, normalize and rationalize patient data from multiple sources to create detailed views of patients. In addition to normalizing real-world data from multiple sources, the OA CDR must be able to analyze discrete data across the unified records. Clinical data is also a collection of structured and unstructured data, from diagnoses to progress notes. To maximize the utility of real-world data, the OA CDR must be able to derive meaning from both data types. For example, a patient with a condition of interest may have a relevant structured diagnosis code, a descriptive pathology result, or a comment in a clinical note. For the connected, comprehensive data to be credible, it must also be free from errors that may be introduced during transmission and aggregation. Ideally, healthcare information is captured as close to the time and place of care delivery as possible, since a single error in capturing data presents risks that can be magnified as the data is transmitted downstream.  This is a key differentiator of the OA CDR:  data is captured during the chemotherapy planning and delivery period with clinical review and decision-making having the impact to both validate the quality of the data being extracted as well as generate clinical observations with respect to disease progression, adverse events, drug toxicity and impact of other co-morbid conditions.

Regulatory Compliance

The OA CDR must protect the security of the data and comply with privacy and reporting regulations. Patient data must be de-identified to protect patient privacy and minimize, if not eliminate, risk of a protected health information breech.  To this end, the OA CDR will be implementing the de-identification standard as defined by section 164.514(a) of the HIPAA Privacy Rule (www.hhs.gov). Under this standard, health information is not individually identifiable if it does not identify an individual and if the covered entity has no reasonable basis to believe it can be used to identify an individual.

Sections 164.514(b) and(c) of the Privacy Rule contain the implementation specifications that a covered entity must follow to meet the de-identification standard. As summarized in Figure 1, the Privacy Rule provides two methods by which health information can be designated as de-identified.

Figure 1. Two methods to achieve de-identification in accordance with the HIPAA Privacy Rule.

The first is the &ldquo;Expert Determination&rdquo; method, where a third-party representative with appropriate knowledge of and experience with generally accepted statistical and scientific principles and methods for rendering information not individually identifiable completes a thorough review of the data store.  Compliant de-identification is achieved if the expert representative determines that the risk is very small that the information could be used, alone or in combination with other reasonably available information, by an anticipated recipient to identify an individual who is a subject of the information; and documents the methods and results of the analysis that justify such determination.

The second is the &ldquo;Safe Harbor&rdquo; method where 18 identifiers of the individual or of relatives, employers, or household members of the individual, are removed.  These identifiers consist of:

Names
All geographic subdivisions smaller than a state, including street address, city, county, precinct, ZIP code, and their equivalent geocodes, except for the initial three digits of the ZIP code if, according to the current publicly available data from the Bureau of the Census:The geographic unit formed by combining all ZIP codes with the same three initial digits contains more than 20,000 people; and
The initial three digits of a ZIP code for all such geographic units containing 20,000 or fewer people is changed to 000

All elements of dates (except year) for dates that are directly related to an individual, including birth date, admission date, discharge date, death date, and all ages over 89 and all elements of dates (including year) indicative of such age, except that such ages and elements may be aggregated into a single category of age 90 or older. Note that because observation and treatment exposure time periods are very important for the types of research to be supported by the OA CDR, dates will be available as a random day interval offset unique to each patient, but not provided to the data recipient.  For example, if patient A gets a random day interval offset of 8, then all dates for any healthcare event for that patient will be shifted forward by 8 days, preserving the treatment and observation period intervals but obfuscating the actual dates that the patient received care.

Satisfying either method would demonstrate that a covered entity has met the standard in &sect;164.514(a) above.  The OA CDR solution will leverage both methods to ensure data is completely de-identified and minimize any risk for re-identification. 

 #SustainabilitySustainability and Scalability

The OA CDR should be able to support a spectrum of business use cases that deliver potential benefits across the entire healthcare ecosystem including pharma, life sciences, payers, providers and patients.  The OA CDR must be adaptable, reusable, standards-based and governed within a sustainable ecosystem.  Because it uses open-source and cloud based big data technologies, the OA CDR will be able to power the natural language processing required to extract, analyze and validate the data, as well as support the ability to train machine learning models for monitoring and detection of patient safety issues such as adverse events and drug toxicity as well as patient opportunities like clinical trial eligibility. 

 #ClinicalReviewClinical Review Integration

A key differentiator of the OA CDR from other real world evidence data sources is that the data ingested is integrated with the additional observation and validation of oncology clinicians that are performing concurrent prior authorization reviews of the requested chemotherapy regimens from the treating physicians.  This enables the engagement of clinicians as close to the time of each encounter (both the treatment planning and the subsequent treatment delivery).  This ability to engage with the care provider and patients concurrently not only yields more accurate data, but also enables full integration of the clinical decision-making process with respect to the planned treatment.  Key data points such as the intent of therapy, reasons for therapy change, patient reported outcomes, etc are more readily collected with the level of engagement that OA clinicians have at the precise point of time care is being planned and immediately after the care plan is delivered or stopped.  Further, this level of engagement may enable future product offerings such as clinical trial eligibility match recommendations.

### Repo/Code/Jobs

16falsedecimallisttrueGenerally, during dashboard development, we collect and transform various data sets into smaller, cleaner intermediate data sets that can then be more easily fed into PowerBI. All of these intermediate data sets need to be developed and hosted in Databricks and should follow some standard best practices. Here&rsquo;s how the code should be written and how it should be stored and structured.

The Repository

Example Repo

All of the code for a dashboard needs to be stored in a sub-repo in the data_analytics_databricks/solutions_reporting/oi_dashboards repository

Your Repo name: /data_analytics_databricks/solutions_repoting/oi_dashboards/{dashboard_name}

Repo Structure:

Notebooks to refresh data sets should sit at the top level of the sub-repo. Each intermediate data set should be refreshed via its own notebook unless there are data dependencies between tables such that it makes sense to process things jointly in one notebook. Breaking things up into separate notebooks is encouraged, however, because modular code is easier to debug and is more resilient to ETL failures (i.e. an error with one intermediate table won&rsquo;t lead to all the tables failing to populate).

Ex: nb_refresh_{dashboard_table_name)

Ex: nb_refresh_{dashboard_table2_name)

schemas: 

__init___.py: An empty file that tells python to view a folder as a python module

table_schemas.py: A file containing the schemas of all of the intermediate tables your dashboard needs. The schemas in this file are used when the tables are first created and are imported into the refresh notebooks and validated before data is written. If a table needs a schema update you must update the schema here, submit a PR, and then write a one off script that DevOps can run in Prod to update the schema on the existing delta table object there (more to follow on the one off script process, see below). This is a control file! If a table&rsquo;s schema does not match the schema in here you will get errors!

nb_build_dashboard_tables: A notebook that creates/initializes empty intermediate tables in the catalog using the corresponding schemas in table_schemas.py. Why create empty tables? I like establishing a table's existence and identity first thing to avoid problems with tables not existing during promotions from environment to environment and also to avoid an initial data insert that contradicts the desired schema.

utils

__init___.py: Empty file that tells python to view a folder as a python module

data_prep_utils.py: A file containing all of the helper functions specific to a given project/dashboard (you can name this file whatever you&rsquo;d like)

jobs

job_build_dashboard_tables.json: A json template that creates the pipeline responsible for creating/initializing all of the intermediate tables.

Note: This is likely just going to be a pipeline with one Task that runs &ldquo;nb_build_dashboard_tables&rdquo;

job_refresh_dashboard_tables.json: A json template that creates the pipeline responsible for refreshing the intermediate data sets. Each data set should have its own task within the pipeline.

Note: If all of your intermediate data sets should be refreshed with the same cadence then you can do them all within one refresh job. However, if the data sets should be updated with different cadences (say if one data set updates quarterly and another updates daily) you should set up one job per cadence. 

scripts

This folder contains any ad-hoc scripts that have been run to update your tables. Need to add a column, update a column name, or change a column&rsquo;s data type? You will need to write a script for DevOps to run in Prod that will make this change without completely tearing down the existing Delta Table and its history. Said script will live here and should be named starting with the update&rsquo;s ADO User Story number.

README.md: Here&rsquo;s where you document the blazes out of your dashboard! 

Coding Requirements & Best Practices

Intermediate data sets need to be stored in the &ldquo;oi_dashboards_{env}&rdquo; catalog. Each dashboard will have a schema that contains all of its intermediate tables

Example: I&rsquo;m a developer creating a dashboard called &ldquo;Sporcle Leaderboard&rdquo; with data sets &ldquo;user_participation_quarterly&rdquo; and &ldquo;user_performance_quarterly.&rdquo; I would store the tables like this:

oi_dashboards_{env}.sporcle_leaderboard.user_participation_quarterly

oi_dashboards_{env}.sporcle_leaderboard.user_performance_quarterly

What does a table_schema.py file look like?

Here&rsquo;s an example with one table. In your file you will create one python object/schema per table. Use the naming convention {table_name}_schema. Think carefully about column types and column order! The primary keys to the table should always appear first (in this example, the table is intended to have one row per case, so OAReferenceNumber is consequently the first column)

Your code must check for duplicates in a dataframe before writing to a table

You&rsquo;ll need to run the dataframe_check_dupes() function for every meaningful partition in your data set. Don&rsquo;t just check for unique rows across all columns; check for unique rows across meaningful sets of columns. Any script that isn&rsquo;t checking for dupes with this function will be rejected in a PR (unless your data is supposed to have duplicates of course!)

Btw it doesn&rsquo;t matter if you&rsquo;re checking for dupes in a different way: your code will still be rejected. You need to use this function (I&rsquo;ve seen too many instances of people checking for duplicates incorrectly and corrupting tables)

Also, of note: You must check for dupes directly before the write step! Don&rsquo;t check earlier and then do a bunch of operations on your dataframe

Your refresh notebooks should not overwrite table schemas

You need to make sure the &ldquo;overwriteSchema&rdquo; option is set to false. Any notebook that doesn&rsquo;t do this will be rejected in a PR.

Your refresh notebooks should validate schemas before writing

I know this sounds redundant because the overWriteSchema option is already required to be false, but in my experience Spark doesn&rsquo;t tell you every way in which a dataframe violates a schema and it doesn&rsquo;t necessarily check column order. The validate_table_schema() function does check column order and will tell you every extra column you have, every column you&rsquo;re missing, and every non-nullable column that contains a Null value (in addition to checking data type mismatches, of course). It will also convert easily convertible data types into the correct data type before writing without throwing an error (i.e. type Long and type Int are the same for values between -2.8b and 2.8 billion, so if your dataframe has type Long and your Table is looking for Int validate_table_schema() will make the conversion for values in that range)

Strings will not be converted to dates and vice versa (even if the strings are in ISO format)

Strings will not be converted to int, long, double, float etc.

Int and Long will not be converted to double or float and vice versa

Double and float are convertible up to 4 decimal places. Any number with more than 4 decimal places will not be converted and will throw an error

When you read in parameter values you should convert them to the correct type

Parameter values are always read into a notebook as type string. Make sure to use the parameter conversion functions to convert them to the correct type before performing any actions on them.

It&rsquo;s a good idea to define the period/dates for each intermediate table

Define the minimum start date and the look back period as job parameters and pass them to the notebook.

It&rsquo;s a good idea to define all the paths of your input data sets as variables at the beginning of your script

Having the input paths clearly defined at the beginning of the script makes it very easy for anyone to open the script and familiarize themselves with the data sets involved. Also, defining the paths as variables will ensure that you only need to update the paths in one place in your script.
