- ⚠️ Gaps &amp; Missing Fields1. Missing eligibility fields that scope determination requires. The docs explicitly require these fields on the Eligibility record, but the schema doesn&#39;t show them:CarrierId (string) — required for identifier-based scope validationAccountId / InsuranceGroupId / BenefitPackage — carve-out identifiersASOIndicator (boolean) — ASO delegation ruleMemberBenefit (enum: Medical / Pharmacy / Medical&#43;Pharmacy)PrescriptionBenefitCarveout (boolean)OONBenefit (code)RiskPatient (flag)EligibilitySource (code)StateOfPlanIssue (state code)MedicareBeneficiaryId / MBI (string, Medicare-only)InsurancePlan / PlanDescription (string)ProductOffering (string/code — finer-grained than Product)PreferredLanguage, AccessibilityCorrespondenceThe CustomFields (json) column on Eligibility could absorb some of these, but that&#39;s a red flag — scope determination rules need to query structured fields, not JSON blobs. Fields like MemberBenefit and ASOIndicator that drive hard business rules must be first-class columns.2. Member contact data is missing. The docs require Address, Phone (home/mobile/work), and Member Representative contact info on the member/eligibility record. None of these appear in Person or Eligibility in the schema.3. No CoverageEffective/CoverageEnd date pair visible on Eligibility — though these are shown in the schema, which is good. ✅ (Actually they are there.)4. BusinessGrain&#39;s FK is PayerVersion (typed Type) — this looks like a data modeling bug. Type is not a valid FK type; it likely should be guid.5. No explicit ProductOffering level. The docs describe a 3-level hierarchy: LOB → Product Line → Product Offering. The schema has MajorLineOfBusiness → Product, but the Product Offering granularity (needed for rule collision prevention per the docs) isn&#39;t clearly represented.6. No MemberRepresentative entity. The docs specify representative name, relationship, address, and phone as required fields from eligibility. This would need its own table or fields on Person/Eligibility. (2421, 681)
- 1. No patient height and weight fields. These are required inputs for every medical dose calculation. Without typed columns, the calculations cannot run or be validated.2. No Treatment Start Date or Treatment End Date on the Case table. Treatment Start Date drives eligibility verification. Treatment End Date drives renewal eligibility. Neither has an explicit field — only a vague Time Due column exists.3. Diagnosis Group is not stored. The requirements say it must be persisted on the case after the International Classification of Diseases version 10 code is selected. Without it, regimen search cannot be filtered and case entry cannot advance.4. No renewal linkage field. The Case table has no reference to a prior case. This is required for auto-approval logic, deviation display, and audit reporting.5. No drug change tracking fields. When a user switches from a non-preferred or branded drug to another, the original drug, the replacement, and the override reason must be recorded for financial impact reporting. No such fields exist.6. Key calculated values are buried in a JSON blob. Allowed dose, maximum billing units, quantity per authorization, cycle count, and benefit type selection for crossover drugs are all likely stored inside the untyped Dosing field. They cannot be queried, validated, or enforced at the database level. These should be explicit columns. (3554, 165)
