Initial Feedback - System Design Document (Data Team)

I am going to share some initial feedback on the System design document based on my review of the Eligibility and Provider sections. I will review the Case Schema once I that has been published.

High-level data flow: I want to make sure that we all have a clear understanding of high-level data flow

There are 3 main systems - Enterprise Data Platform (EDP), Transactional System Repository (EligibilityMS, ProviderMS, Drugs MS etc), Provider Data in Case Management (newUM).
- The Client Data flows into EDP.
- EDP owns the pipeline to populate the Provider data to be used for Case Management into ProviderMS.
- newUM queries ProviderMS to get the Provider Data that needs to be attached as Ordering and Requesting Providers when creating a case.
- newUM sends the case data along with the provider information to EDP.
- EDP is responsible for merging newUM data along with MATIS data and SmartUM data into data warehouse tables that feed Analytics Services.

[Provider data flow diagram image]

This above understanding will be crucial to the re-use of the existing Services - we are not redesigning Eligibility Service, Drug Service etc. Provider Service will be a re-design of the transactional repository schema (ProviderMS) and the API to search / retrieve Provider data as a part of Case creation and execute network checks.

Common understanding on the foundational topics:
- Socialize the Data Flow design above.
- Discuss the section - "Data Exchange: NewUM → Databricks Bronze"
  - Why do we need a medallion layer architecture for newUM Data?
  - The DE team will be integrating the newUM data with MATIS and SmartUM for Analytics Services.
- Discuss how the Unity Catalog structure will be used to provide clarity on newUM Schema in Databricks.
  - The decision on using Event Hubs and how to make sure that the JSON schema is well defined and controlled for the DE team to consume these events and recreate the transactional data will be crucial.
- Marc Gale - We have not implemented event sourcing data exchange strategy and this will be a significant lift for the data team. Unless we learn otherwise in the design discussions.
- The sections on - Metrics, Monitoring & Alerting and Deployment, Integration and Rollout Sections - can be follow up discussions.

Provider Feedback:
- Payer-Specific Network Status by Specialty – How is the data model answering - "Provider X is in Network Y under Payer Z for Specialty S."
- Location-Specific Participation: How is this model answering "Provider participates in Payer A's network at Location B with Specialty C."
- Office Staff Management – How do we model the relationship between Office Staff (users of Provider portal and newUM) and the Providers (ordering / servicing)? Should this be managed outside the core provider data model. If so, how? This will be crucial to SSO and direct web user access management.
- Manual Provider Data Entry & Merge – How are we accounting for having manual data entry be captured, and merged into the main data tables?
- This data model represents ProviderMS (the repository of all providers that are Payer-sourced or manually entered). Where is the Provider Data Model attached to a case (Ordering Provider / Servicing Provider)?
- Why is this model very different compared to the data model shared by the Data Engineering team? What challenges are addressed by this new data model that was a gap in the DE data model?

Eligibility Feedback:
- Why are these tables very different from EligibilityMS? Is this the Data model for the Case Management System to handle Eligibility and the configurability of the system using Payer hierarchy?
- Payer Hierarchy Chain --> Why do we not have the concept of "Parent Payer Org"
- Plan <> Network <> Provider junction — crucial for Member Eligibility Check, Provider Network Check, Formulary Benefit Checks
- What is the "grain" of the Eligibility table — Typically, MemberID, PayerID, Group+PlanID, Effective Date is the grain.
- "Payer file inconsistency creates ambiguous member status" → why is this a newUM Problem? Should be handled by DE Pipeline.

--- Message: Michal Mucha, 3/31 6:47 AM ---
Venky just to keep you in a loop, after yesterday's meeting, I feel we need to carry on with investigation, at the moment we have 3 strong candidate to migrate data form Databricks to NewUM Postgress:

1. Lakebase Synced Tables — Databricks-native managed sync from Delta/Iceberg directly into a Lakebase-hosted PostgreSQL-compatible instance — no pipeline code to write
2. ADLS Gen2 Staging + Airflow + PostgreSQL COPY — Most mature and understood option - Databricks writes Parquet to ADLS, PostgreSQL bulk-loads via COPY, shadow table swap ensures zero reader disruption
3. Apache Iceberg → PostgreSQL — Most architecturally elegant - Databricks writes Iceberg natively (one config change), pg_lake lets PostgreSQL query it directly with no ingestion step, queries run 10x+ faster via DuckDB, Eliminates the COPY pipeline, the Airflow DAG, and the swap pattern entirely; object storage costs ~10x less per GB than Postgres block storage

More details you can find here NewUM Data System Design.docx
Data exchange mechanism – work in progress chapter
We need to investigate more over upcoming sprint those 3 approaches.

Providers Domain — Recent Decisions & Open Questions

Network Participation
- Two separate bridge tables: one for Practitioners (T1), one for Organizations (T2).
- Practitioner's network status is independent of organization's network status.
- Network tier is V2 item — not required for V1.

Out of Network:
- Search is cross-health-plan in V1.
- No provider-on-the-fly in V1.
- OON network check/determination: Out of scope — R2.
- Manual provider entry: Out of scope — R2.
- Provider network data ingested into NewUM: Out of scope — R2.
- Rules engine receives INN/OON as input: Planned but null/unavailable in V1.
- No V1 requirements define the OON workflow.

[QUESTION] Inna/Alison: When VerifyIfProviderIsInNetwork = True fires and servicing provider is OON, what is the expected case behavior in V1?

Provider Address Model
- Shared Address dimension table, append-only (never updated/deleted).
- Path 1 (Regular Providers): T1 → T1T2_Link → T2 → Address
- Path 2 (Sole Proprietors): T1 → Address directly
- Address types (mailing, practice, billing) via bridge table field.
- Case-level address captured at assignment time, decoupled from future changes.

[QUESTION] Inna/Alison: Sole proprietor flag — how determined at ingestion time?

Office Contact at Case Level
- Office contact (phone, fax, email, address) lives at case level, not provider level.
- Address pre-populated from provider data; contacts manually entered per case.
- Two records per case (ordering + treating) with provider_type enum.
- Case-level copy must NOT be overwritten by subsequent ETL updates.

[QUESTION] Joseph Destefano, Jack Hall: Auditability to be discussed for this table.

Organization Affiliation in Search Results
- Product requires org affiliation name display. Refers to T2 entity.
- T1–T2 linkage currently not working in Mattis.
- Users make independent selections for ordering, treating, place of service.

Place of Service Code
- 2-digit CMS-standard code captured at case level.
- Payer-configurable (different payers restrict available values).

[QUESTION] Kim Jewell: Is this on configuration team's radar?

Facility Type
- Payer-specific configurable dropdown, NOT global hardcoded list.
- Another item for configuration team.

Provider Schema — T1/T2 Renaming
- DECISION: Adopting FHIR standard terminology: Practitioner (T1), Organization (T2).
- CMS mandate moving toward FHIR APIs for provider data.

[QUESTION] Kim Jewell: Where should payer configuration live? Feels like Payer domain.

Manual Provider Entry and Out-of-Network Providers
Three use cases:
1. Provider found in ETL, used as-is.
2. Provider found but requires corrections — override at case level.
3. Provider not found — created from scratch.

AGREED DIRECTION: Case-level snapshots.
- Snapshot taken when provider assigned to case.
- Snapshot can be modified independently of source.
- Source provider tables remain read-only (ETL only writes).
- Once snapshot taken, NOT overwritten by subsequent ETL updates.

[QUESTION] Inna/Alison: How often is manually entered provider truly new vs. office staff conflated with provider?

Case-Level Data Never Written Back to Provider Master Tables — FIRM design principle in V1 and R2.

Retro Case / Time Travel Problem
- Cases sometimes submitted retroactively.
- System must evaluate against data valid at original service date, not today's.
- No clear mechanism currently exists for this.

[QUESTION] Alison/Inna: Can the system process retro cases as if time-traveled?

Auditability Pattern
- FAVORED DIRECTION: Snapshot approach (copy provider data at case level).
- Remaining gap: retro network checks (network participation data changes over time).
