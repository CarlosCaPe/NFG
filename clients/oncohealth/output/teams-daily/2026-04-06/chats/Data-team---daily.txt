 : Hey Michal Mucha, Cory Johnson , I hope this short explanation resolves the questions raised during the daily. The full investigation document has more detail, but I think for now the investigation is complete and I can answer these. Of course this still requires a code review, but basically this is the main idea:

Q1: Why Iceberg REST if the data is already in Delta Lake? Why not just read Delta directly?

 

Delta Lake already has ACID transactions, schema evolution, and time travel. But the problem is when something outside Databricks needs to read that data.

 

Delta is a format that only Spark and a few libraries (delta-rs) speak well. If our .NET backend, Snowflake, or Trino needs to read a table, they have two options:

 

Option A — Read Delta directly:

You have to give them the keys to Azure storage (ADLS). Keys you generate, rotate, and revoke.
If they read while Databricks is writing, they can see corrupted or partial data.
They bypass Unity Catalog entirely — no record of who read what, no column filters, no row-level permissions.
They must understand Delta's internal format (parse _delta_log/, handle deletion vectors, column mapping). Most can't.

Option B — Iceberg REST (recommended):

The external system does a GET to a URL. The API gives it a temporary 1-hour token, valid only for the tables it has permission to see.
All reads go through Unity Catalog — audit log, permissions, masks all apply.
No need to understand Delta internally. Iceberg is an open standard that PyIceberg, Spark, Snowflake, Trino, DuckDB, and Flink all speak natively.
No data duplication. UniForm generates Iceberg metadata that points at the same Parquet files Delta already uses.

Analogy: Delta is the internal language of our data warehouse. Iceberg REST is a translator that lets any external system read it — without learning Delta, without us handing over the warehouse keys, and with a full record of everything they read.

Q2: Is the cost really zero? The API must have some cost.

 

What costs $0:

API license — none. Comes free with Databricks.
Per-call charge — none. Not like Azure API Management which charges per request.
Service principal — free Entra ID feature.
Temporary SAS tokens — generated by the API itself, no cost.

What we already pay for (doesn't change):

The Databricks workspace — already running, already billed.
The clusters doing daily ingestion from 9 data sources — already running.
ADLS storage — already provisioned.

Real incremental costs (based on our inventory: 127 Delta tables, 6 catalogs, daily ingestion, East US 2):

Extra driver memory (+5-15% for Iceberg metadata): $0 if clusters have headroom, $15-30/mo if we need to bump the instance type
Iceberg metadata files on ADLS: < $0.01/mo (19 MB/month)
ADLS read transactions (GET ops): ~$0.07/mo (assuming 50 queries/day)
Network egress: $0 same-region, $0.087/GB if cross-region

Possible additional costs:

Private Link (if InfoSec requires it): +$7.30/mo
SQL Warehouse for reads: $22/hr when active (PRO Small)
Runtime upgrade (if < DBR 14.3 LTS): $0 but needs testing

Realistic monthly total:

Best case (clusters have headroom, same region): < $1/mo
Typical (driver bump needed): $15-30/mo
With Private Link: $22-37/mo

One-time setup: ~5-8 hours of data engineer time (enable UniForm, permissions, test with PyIceberg).
