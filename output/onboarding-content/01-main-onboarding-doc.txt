Project introduction
Project Purpose and Background
The purpose of the NewUM project is to replace the end-of-life MATIS core monolith with a modern, configurable, and high-performance Utilization Management (UM) case management engine. The current MATIS system is based on a limiting code-based approach that prevents efficient customization, rapid client onboarding, and new feature development. NewUM will drive operational efficiency and quality across OncoHealth by modernizing architecture and data modeling.
Project Goals and Objectives
* Improve client experience through better analytics (operational reporting and value-added insights for customers and internal OncoHealth teams) and provide a more responsive client service experience when making needed changes
* Improve the health care provider experience with a more responsive Portal and extensibility of drug-based approval logic into provider-facing tools
* Create an invisible migration experience for health plans and providers such that newUM effort is perceived as an ‘upgrade’ of the system
* Improve OncoHealth Technical Team Efficiency by moving to configurable architecture and a new modern monolith
* Improve OncoHealth Case Processing Team efficiency (clinical and non-clinical) by replacing the underlying abstraction from the MATIS platform  (regimen-based approvals) to drug-based approvals (potentially while retaining the concept of the regimen) and significantly improving the UI experience with a cleaner set of clicks
* Create clear ‘hooks’ for AI agents to engage in workflow through design of APIs and UI that are well-suited to current levels of agentic AI capability


Additional considerations:
The NewUM platform will provide a configuration-based approach to workflow creation and will deliver:
* Client-based configuration rather than custom product coding to enable workflow variations and decision rules for new and existing health plan customers
* Enhanced clinician experience for internal users (e.g., see case history and more easily see other cases for the same members) and external users interacting with the OncoHealth portal[a]
* Trailing data guardrails and manual data repair integrated into the new workflow config and UI[b]
* A workflow platform that enables integration with AI agents throughout the UM process
* A redesigned user experience that addresses the foundational issues identified in ‘20251107_OncoHealth Initial Research Results.docx’. (In addition, we will identify places in the core redesigned workflows where AI can provide productivity gains.)
* Provider registration directly in the platform
* A consistent, fully modern Enterprise Data Platform (EDP) for ingestion and reporting
In scope
* Replacement of MATIS core with a new configurable case management engine.
* Redesigned User Interface (UI) for case entry, clinical review, and decision-making.
* Development of a continuous Migration Pipeline for historical data.
* Integration with existing peripheral services for OneUM  (Letters V2, Admin Portal, D&A/Databricks).
* Implementation of post-launch features including Appeals Case Management and Turn-around-Time (TaT) reporting (to be confirmed for Phase 6).[c][d][e]
Out of scope
* SmartUM Client Migrations (requires separate strategic project).
* Advanced Provider Mastering (3rd party PDM service interfaces).
* Interoperability APIs for FHIR/CMS-0057 compliance.
* Native AI-based Clinical Extraction (to be handled via external partnership).
Success criteria
* Faster implementation cycles for new health plans (~60 days down from ~90)
* Better metrics than SmartUM (e.g., case entry per person per month)[f][g]
* Workflow extensibility (e.g., delegated appeals workflows can be created in a consistent and straightforward manner)
* Faster employee onboarding
* Lower training costs and turnover
* More efficient operations and more favorable staffing ratios
* Increased portal adoption by providers
* Reduced operational load and compliance risk
* Simplified D&A (ingress and compliance/reporting)
* Reduced case delays and pressure on TaT
There will be a separate governance document with exact metrics, baselines and monitoring.

Risk management
Instead of tracking risks, blockers, and decisions across various emails and meeting notes, we will be using a RAID log (newUM_RAID.xlsx) as our single source of truth for everything that could impact our delivery.
Moving forward, everyone should use the log for the following:
* R (Risks): Identify potential "what-ifs." If you see a storm brewing (e.g., a potential tech bottleneck), log it so we can mitigate it early.
* A (Assumptions): Document what we’re taking for granted. If you're building based on the belief that "API access will be ready by Friday," put it here.
* I (Issues): Log active blockers. If something is broken or delayed right now, it goes here with an owner assigned to fix it.
* D (Dependencies/Decisions): Record what we are waiting on from other teams, and document key decisions so we don't have to ask "Why did we decide that?" three months from now.

High-Level Architecture[i]
Newfire Miro (if you don’t have access, ask for it, slack Rachel Collier). NOTE: Please do not create a new board within Miro at this time. We are limited to 3 boards and if a new board is created it will make another board View Only. We are working with OncoHealth on this limitation. All work should go within the NewUM board. The architecture is currently in a draft state; please ask questions about it! Poke holes if you can. The most uncertainty is currently (as of 12/22/25) around:

* case data model
* management of business rules
* frontend configurability
* reporting pipelines


The backend architecture is split into three conceptual subsystems: Rules, Data and Interface. 


The Rules Subsystem will manage, compile and run sets and versions of rules. It will offer a CLI-based development environment for configuration specialists and interact with the Interface subsystem via Azure Service Bus. It will use the data access layer as a library to interact with cases. 


The Data Subsystem will ensure consistency of data throughout NewUM. All access to transactional data should go through the data access layer. It will offer a bulk ingest API for migration and ingest pipelines, or it may be used as a library in other subsystems. All reporting needs should be handled with an OLAP database specifically for reporting, populated by the data access layer.  


The Interface Subsystem will offer a unified interface to the UI and also peripheral services from OneUM. 


Our initial development will be focused on a vertical slice of the system that can create, view and update a case as well as attach an eligibility record. The first slice should also be able to modify case state using business rules. We will start with the green-highlighted components in the Miro draft architectural diagram.
The data access layer is one of the most complex parts of the core. The technical demands for tracking cases are extensive: [j]

* data for each case is spread across multiple tables
* some payers will want custom fields coupled with custom business rules
* maintain a complete audit log for each case, preferably with a non-PHI view for overseas devs
* case data must be kept for years[k]
* data will come from multiple sources: UI, external comms, ingest, Matis migration, etc.
* MS SQL database (OADB) is under strain in current system, while the business needs to grow
* a case should enforce one writer at a time to prevent race conditions

The other schemas also have unique needs. The eligibility and provider schemas will need some degree of mastering to avoid confusion from duplicate provider and member records. 


Leaving the core, NewUM will need to integrate with several peripheral services from OneUM like a letter-sending system, a fax system, a document manager and so forth. Most of these other services need upgrades (particularly for auth) and will require scaffolding so that the old and new system can coexist. We should use this .NET package for consistency with other OncoHealth systems unless there is a good reason not to.

Coding standards
FE TEAM
 Architecture & Project Structure[ag]
      * Feature or page based - Folders: Group files by functionality rather than by type. Each folder should contain its own components, hooks, and tests.
      * Flat Component Hierarchy: Keep components small. If a component grows large, break it into smaller sub-components.
      * Barrel Files: Use index.ts files in folders to simplify imports
      * Path Aliases: Use Vite path aliases (e.g., @/components/*) to avoid "import hell" .
React & TypeScript Standards
      * Functional Components: Use arrow functions (const Component = () => ...) for consistency.
      * TypeScript Strict Mode: Enable strict: true in tsconfig.json. No any, for edgecases use unknown
      * Potentially we will use the interface only for  APIs/libraries and type aliases for internal component props and utils
      * Hooks Logic: Move complex logic into custom hooks or utils  to keep the UI layer (JSX) clean and declarative.
Code Quality (ESLint & Prettier)
      * Pre-commit Hooks: Use Husky or smth similar and lint-staged to run ESLint and Vitest on changed files before every commit.
      * Import Sorting: Use a plugin to group imports: 1. React/External 2. Internal/Features 3. Styles/Assets.
      * Use Prettier for auto formatting
      * Use Eslint to track code style issues
AI Assisted Coding
Agentic coding and AI tooling is very very strongly encouraged.

      * Agnetic coding (Claud Code is provisioned.  Cursor and others on request) shall be used to assist design, development, testing and pull request reviews.
      * The used tools should be disclosed to the project leadership

      * Knowledge sharing sessions on AI-assisted and agentic coding should be held by team leads frequently
      * Once you’ve aligned on an AI coding tool with your team leader, please have your manager submit a license request via the Newfire IT Support Portal.

Business Analysis Approach
      * BA defines case workflows, configuration behaviors, rules intent, and data expectations for NewUM
      * BA work starts in Phase 1 and continues through build, migration, and hardening phases
      * BA operates ahead of engineering, ensuring backlog items are ready before development begins
      * BA does not own product prioritization or technical design decisions

Business Analysis Governance
         * Business Analysts own requirements clarity, completeness, traceability, and readiness
         * Product Management owns scope, sequencing, and business priority
         * Architecture and Technical Leads own solution design and implementation approach
         * No requirement enters development without Business Analyst readiness confirmation
         * Requirements decisions and approvals are documented and traceable
